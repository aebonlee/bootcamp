## 4.1 Stable Diffusion 실습 개요

### (1) Stable Diffusion의 작동 원리 요약 (텍스트 → 잠재공간 → 이미지)

Stable Diffusion은 세 가지 핵심 신경망 구성 요소가 상호 협력하여 텍스트에서 이미지를 생성하는 잠재 확산 모델(Latent Diffusion Model, LDM)입니다. 픽셀 공간에서 직접 확산(diffusion)을 수행하는 기존 방식과 달리, 압축된 잠재 공간(latent space)에서 확산을 수행하기 때문에 계산 효율이 훨씬 뛰어납니다.

전체 생성 과정은 크게 세 단계로 나눌 수 있습니다.

| 구성 요소 | 역할 | 세부 설명 |
| --- | --- | --- |
| CLIP Text Encoder | 텍스트 → 임베딩 벡터 | 입력 프롬프트를 토큰화하고 768/1024차원의 의미 벡터로 변환. 이 벡터가 UNet의 Cross-Attention을 통해 이미지 생성을 유도함 |
| UNet (Denoising Network) | 노이즈 예측 및 제거 | 잠재 벡터에 추가된 가우시안 노이즈를 단계적으로 제거하는 역할. 텍스트 임베딩과 Cross-Attention으로 결합되어 텍스트 조건에 맞는 이미지를 생성 |
| VAE (Variational AutoEncoder) | 잠재공간 ↔ 픽셀공간 변환 | Encoder: 실제 이미지 → 잠재 벡터 (4×64×64). Decoder: 최종 잠재 벡터 → 실제 이미지 픽셀 (3×512×512) |
| Scheduler (DDPM/DDIM 등) | 확산 스텝 제어 | 몇 번의 역확산 스텝으로 노이즈를 제거할지 결정. DDIM은 20~50 스텝으로 고품질 이미지 생성 가능 |

생성 흐름을 순서대로 따라가면 다음과 같습니다.

#### Stable Diffusion 이미지 생성 단계별 흐름

1. **텍스트 인코딩**: 프롬프트 문자열 → CLIP Text Encoder → 텍스트 임베딩 벡터 (shape: `[1, 77, 768]`)
2. **초기 노이즈 생성**: 랜덤 가우시안 노이즈로 잠재 벡터 초기화 (shape: `[1, 4, 64, 64]`)
3. **역확산 루프 (Denoising Loop, 기본 50 스텝)**:
   - UNet이 현재 잠재 벡터의 노이즈 성분 예측
   - Scheduler가 예측된 노이즈를 바탕으로 잠재 벡터 업데이트
   - 텍스트 임베딩이 Cross-Attention을 통해 각 스텝을 안내
4. **VAE 디코딩**: 최종 정제된 잠재 벡터 → VAE Decoder → 512×512 픽셀 이미지 (RGB)
5. **후처리**: [0, 1] 범위 정규화 → `PIL.Image` 변환 → PNG/JPEG 저장

Classifier-Free Guidance(CFG)는 이미지 품질과 프롬프트 충실도를 조절하는 핵심 파라미터입니다. CFG 스케일(`guidance_scale`)이 높을수록 프롬프트와 더 가깝지만 다양성이 줄어들고, 낮을수록 자유롭지만 프롬프트에서 벗어날 수 있습니다.

| guidance_scale 값 | 특성 | 권장 사용 상황 |
| --- | --- | --- |
| 1.0 ~ 3.0 | 프롬프트 충실도 낮음, 매우 다양한 결과 | 창의적 실험, 예술적 자유 |
| 5.0 ~ 8.0 | 균형잡힌 품질, 중간 충실도 | 일반적 생성, 가장 많이 사용 |
| 9.0 ~ 12.0 | 높은 충실도, 선명하고 정확한 결과 | 정밀한 프롬프트 묘사가 필요할 때 |
| 13.0 이상 | 과도한 충실도, 색상 왜곡 가능 | 특수한 경우에만 사용 권장 |

### (2) diffusers 라이브러리 소개 및 설치

diffusers는 Hugging Face가 개발한 확산 모델(Diffusion Model) 전용 라이브러리입니다. Stable Diffusion 외에도 DALL-E 2, Imagen, DeepFloyd IF 등 다양한 생성 모델을 일관된 인터페이스로 사용할 수 있습니다.

| 구성 모듈 | 설명 |
| --- | --- |
| `StableDiffusionPipeline` | 기본 txt2img 파이프라인. 프롬프트 → 이미지 생성 |
| `StableDiffusionImg2ImgPipeline` | 이미지+프롬프트로 이미지 변형 (img2img) |
| `StableDiffusionInpaintPipeline` | 마스크 영역만 재생성 (인페인팅) |
| `StableDiffusionControlNetPipeline` | ControlNet 조건을 결합한 구조 보존 생성 |
| `StableDiffusionXLPipeline` | SDXL: 1024×1024 고해상도 생성 지원 |
| `AutoPipelineForText2Image` | 모델 구조를 자동 감지하여 적절한 파이프라인 선택 |
| `DiffusionPipeline.from_pretrained()` | 모든 확산 모델 공통 로드 메서드 |

설치는 pip 한 줄로 완료됩니다. 다만 버전 간 호환성 문제가 발생할 수 있으므로 아래의 검증된 버전 조합을 사용하길 권장합니다.

```bash
# ── 기본 설치 ────────────────────────────────────────────
# diffusers: 확산 모델 핵심 라이브러리 / transformers: CLIP 텍스트 인코더 등 제공 / accelerate: 장치 배치·오프로드 지원
# 버전을 고정(==)하는 이유: 라이브러리 간 API 변경으로 인한 호환성 오류를 예방하기 위함
pip install diffusers==0.27.2 transformers==4.40.1 accelerate==0.29.3
# torch/torchvision는 CUDA 12.1 빌드를 명시적으로 받는다(--index-url로 GPU용 휠 지정). 일반 pip은 CPU 버전이 깔릴 수 있음
pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121

# ── 선택적 추가 패키지 ────────────────────────────────────
# xFormers: Attention 연산 최적화 (VRAM ~30% 절감)
# 메모리 효율적 attention 커널을 제공 → 같은 GPU로 더 큰 이미지/배치 처리 가능
pip install xformers==0.0.25

# ControlNet, img2img 실습용
# opencv-python-headless: GUI 의존성 없는 OpenCV(서버용), Pillow: PIL 이미지 입출력
pip install opencv-python-headless Pillow

# API 서버 구축용
# fastapi: 웹 프레임워크 / uvicorn: ASGI 서버 / python-multipart: 파일 업로드(form-data) 파싱
pip install fastapi uvicorn python-multipart

# ── 설치 확인 ────────────────────────────────────────────
# 파이썬에서 diffusers를 임포트해 버전을 출력 → 정상 설치 여부를 한 줄로 검증
python -c "import diffusers; print(diffusers.__version__)"
# 출력: 0.27.2
```

### (3) 모델 로딩: CompVis/stable-diffusion-v1-4, stabilityai/stable-diffusion-2 및 Hugging Face 토큰 발급

Stable Diffusion 모델을 처음 로드할 때는 Hugging Face Hub에서 수 GB의 모델 파일을 다운로드합니다. 일부 모델은 라이선스 동의가 필요하며, 이 경우 Hugging Face 액세스 토큰(Access Token)이 요구됩니다.

#### Hugging Face 액세스 토큰 발급 방법

1. https://huggingface.co 에 접속하여 계정을 생성하거나 로그인합니다.
2. 우측 상단 프로필 아이콘 → Settings → Access Tokens 메뉴로 이동합니다.
3. [New token] 버튼을 클릭하고 토큰 이름을 입력합니다. Role은 read로 설정합니다.
4. 생성된 토큰(`hf_`로 시작하는 문자열)을 복사하여 안전한 곳에 저장합니다.
5. 모델 페이지(예: `CompVis/stable-diffusion-v1-4`)에 접속하여 사용 약관(License Agreement)에 동의합니다.

주요 Stable Diffusion 모델별 특징을 비교합니다.

| 모델 ID | 버전 | 해상도 | VRAM 요구 | 특징 |
| --- | --- | --- | --- | --- |
| `CompVis/stable-diffusion-v1-4` | SD 1.4 | 512×512 | 4GB+ | 최초 공개 버전, 가장 광범위한 파인튜닝 모델 호환 |
| `runwayml/stable-diffusion-v1-5` | SD 1.5 | 512×512 | 4GB+ | SD 1.4 개선판, 현재 가장 널리 사용되는 기본 모델 |
| `stabilityai/stable-diffusion-2` | SD 2.0 | 512×512 | 6GB+ | OpenCLIP 채용, 768 해상도 지원 버전도 별도 제공 |
| `stabilityai/stable-diffusion-2-1` | SD 2.1 | 768×768 | 8GB+ | SD 2.0 개선, 파인튜닝 친화적 |
| `stabilityai/stable-diffusion-xl-base-1.0` | SDXL | 1024×1024 | 10GB+ | 이중 UNet 구조, 최고 품질. 고사양 GPU 필요 |

```python
import torch
# StableDiffusionPipeline: SD 1.x 전용 / DiffusionPipeline: 모델 구조를 자동 감지하는 범용 로더
from diffusers import StableDiffusionPipeline, DiffusionPipeline

# ── SD v1.5 로딩 (가장 범용적) ──────────────────────────
model_id_v15 = "runwayml/stable-diffusion-v1-5"  # Hub의 모델 식별자(저장소 경로)

# from_pretrained: Hub에서 가중치를 내려받아 파이프라인(텍스트인코더+UNet+VAE+스케줄러)을 한 번에 구성
pipe_v15 = StableDiffusionPipeline.from_pretrained(
    model_id_v15,
    torch_dtype=torch.float16,    # fp16으로 VRAM 절감 (fp32 대비 메모리 절반)
    use_safetensors=True,         # safetensors 형식 우선 로드 (보안, 속도)
    safety_checker=None,          # 안전 필터 비활성 (연구/실습용)
    requires_safety_checker=False # 안전 필터가 없어도 경고를 띄우지 않도록 설정
)
pipe_v15 = pipe_v15.to("cuda")    # 모든 구성 요소를 GPU 메모리로 이동(실제 추론은 GPU에서)

# ── SD v2 로딩 ───────────────────────────────────────────
model_id_v2 = "stabilityai/stable-diffusion-2"

# SD2는 OpenCLIP 등 구조가 달라 범용 DiffusionPipeline으로 로드(적절한 클래스를 자동 선택)
pipe_v2 = DiffusionPipeline.from_pretrained(
    model_id_v2,
    torch_dtype=torch.float16,
    use_safetensors=True
).to("cuda")

# ── SDXL 로딩 (고사양 GPU 필요) ─────────────────────────
model_id_xl = "stabilityai/stable-diffusion-xl-base-1.0"

pipe_xl = DiffusionPipeline.from_pretrained(
    model_id_xl,
    torch_dtype=torch.float16,
    use_safetensors=True,
    variant="fp16"  # fp16 전용 가중치 파일을 받아 다운로드 용량·VRAM을 절약
).to("cuda")

print("모든 모델 로드 완료")
```

> **[코드 4-3]** SD v1.5, v2, SDXL 모델 로딩 비교

> `use_safetensors=True` 옵션을 반드시 사용하세요. safetensors 형식은 기존 `.bin` 형식보다 로드 속도가 빠르고, 악성 코드가 포함된 pickle 파일을 방지하는 보안 이점이 있습니다.

### (4) CPU vs GPU 성능 비교 (Colab / 로컬)

Stable Diffusion은 GPU 없이 CPU에서도 실행할 수 있지만 속도 차이가 극단적입니다. 아래 표는 512×512 이미지 1장을 50 스텝으로 생성할 때의 실측 시간입니다.

| 실행 환경 | 장치 | 정밀도 | 1장 생성 시간 | 비고 |
| --- | --- | --- | --- | --- |
| 일반 PC (로컬) | Intel Core i9 CPU | fp32 | 약 25~40분 | 실용적 사용 불가 수준 |
| 구글 Colab 무료 | NVIDIA T4 GPU 15GB | fp16 | 약 15~25초 | 무료, 연결 시간 제한 있음 |
| 구글 Colab Pro | NVIDIA A100 GPU 40GB | fp16 | 약 4~8초 | 고품질 실습 환경 |
| 로컬 워크스테이션 | NVIDIA RTX 3060 12GB | fp16 | 약 10~18초 | 가성비 좋은 개인 환경 |
| 로컬 워크스테이션 | NVIDIA RTX 4090 24GB | fp16 | 약 2~5초 | 최고 성능 개인 환경 |
| 클라우드 서버 | NVIDIA A10G 24GB (AWS) | fp16 | 약 6~12초 | 프로덕션 배포 적합 |

```python
# 실행 환경 자동 감지 및 최적 설정
import torch
from diffusers import StableDiffusionPipeline

def load_pipeline_for_env(model_id: str):
    """실행 환경에 따라 자동으로 최적 설정으로 파이프라인 로드"""
    if torch.cuda.is_available():  # NVIDIA GPU(CUDA) 사용 가능 여부 확인
        # GPU 사용 가능: fp16으로 로드
        # 0번 GPU의 전체 메모리(byte)를 GB로 환산 → 최적화 수준 결정에 사용
        vram_gb = torch.cuda.get_device_properties(0).total_memory / 1e9
        print(f"GPU 감지: {torch.cuda.get_device_name(0)} ({vram_gb:.1f} GB VRAM)")
        pipe = StableDiffusionPipeline.from_pretrained(
            model_id,
            torch_dtype=torch.float16,  # GPU에서는 fp16이 메모리·속도 모두 유리
            use_safetensors=True,
            safety_checker=None,
            requires_safety_checker=False
        ).to("cuda")

        if vram_gb < 8:
            # VRAM이 8GB 미만이면 추가 메모리 최적화
            # attention slicing: attention 계산을 잘게 나눠 VRAM 사용 피크를 낮춤(약간 느려짐)
            pipe.enable_attention_slicing()
            print("낮은 VRAM 환경: attention slicing 활성화")
    elif hasattr(torch.backends, "mps") and torch.backends.mps.is_available():
        # Apple Silicon (M1/M2/M3) — macOS의 Metal GPU 백엔드(MPS) 사용
        print("Apple Silicon MPS 감지")
        pipe = StableDiffusionPipeline.from_pretrained(
            model_id,
            torch_dtype=torch.float32,  # MPS는 fp16 미지원 → fp32로 로드
            use_safetensors=True,
            safety_checker=None,
            requires_safety_checker=False
        ).to("mps")
    else:
        # CPU 폴백 — GPU가 전혀 없을 때 (속도가 매우 느림)
        print("경고: GPU 없음. CPU 모드로 실행합니다 (매우 느림).")
        pipe = StableDiffusionPipeline.from_pretrained(
            model_id,
            torch_dtype=torch.float32,  # CPU는 fp16 연산이 비효율적 → fp32 사용
            use_safetensors=True,
            safety_checker=None,
            requires_safety_checker=False
        )  # .to(...) 호출 없음 → 기본 장치(CPU)에 유지
    return pipe  # 환경에 맞게 구성된 파이프라인 반환

pipe = load_pipeline_for_env("runwayml/stable-diffusion-v1-5")
```

### (5) 첫 이미지 생성 실습: 텍스트 → PNG 파일 저장

환경 설정이 완료되었으면 이제 첫 번째 이미지를 생성해 봅니다. Stable Diffusion으로 좋은 이미지를 생성하려면 프롬프트를 영어로 상세하게 작성하는 것이 중요합니다.

#### ✍ 효과적인 프롬프트 작성 원칙

- **긍정 프롬프트(positive prompt)**: 원하는 내용을 구체적으로 서술
  - 예) `"a majestic white lion in a mystical forest, golden hour lighting, 8K, photorealistic"`
- **부정 프롬프트(negative prompt)**: 원하지 않는 요소를 제거
  - 예) `"blurry, low quality, watermark, text, deformed, ugly, bad anatomy"`
- **스타일 키워드 추가**: `"4K"`, `"8K"`, `"photorealistic"`, `"oil painting"`, `"anime style"`, `"cinematic"`
- **아티스트 참조**: `"in the style of Van Gogh"`, `"Greg Rutkowski style"` 등

```python
import torch
from diffusers import StableDiffusionPipeline
from PIL import Image  # 생성 결과는 PIL.Image 객체로 반환됨
import os
import time

# ── 파이프라인 로드 ──────────────────────────────────────
pipe = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
).to("cuda")

# ── 프롬프트 설정 ─────────────────────────────────────────
# 긍정 프롬프트: 생성하고 싶은 내용을 구체적으로(피사체+조명+화질+스타일 순으로 나열하면 효과적)
prompt = (
    "a majestic white lion resting in a mystical ancient forest, "
    "golden hour lighting, ultra detailed fur, 8K resolution, "
    "photorealistic, award-winning photography, cinematic composition"
)
# 부정 프롬프트: 결과에서 제거하고 싶은 요소(흐림·워터마크·기형 등)를 나열
negative_prompt = (
    "blurry, low quality, watermark, text, deformed, ugly, "
    "bad anatomy, out of frame, extra limbs, duplicate"
)

# ── 이미지 생성 ──────────────────────────────────────────
start = time.time()  # 생성 소요 시간 측정 시작

with torch.inference_mode():  # 그래디언트 계산 완전 비활성화 → 메모리 절약·속도 향상
    result = pipe(
        prompt=prompt,
        negative_prompt=negative_prompt,
        width=512,
        height=512,
        num_inference_steps=50,   # 역확산 스텝 수 (20~50 권장). 많을수록 정밀하지만 느림
        guidance_scale=7.5,       # CFG 스케일 (7~9 권장). 높을수록 프롬프트 충실, 낮을수록 다양
        num_images_per_prompt=1,  # 한 번에 생성할 이미지 수
        generator=torch.Generator("cuda").manual_seed(42)  # 시드 고정 → 같은 결과 재현 가능
    )

elapsed = time.time() - start  # 경과 시간 계산
print(f"생성 시간: {elapsed:.2f}초")

# ── 이미지 저장 ──────────────────────────────────────────
os.makedirs("./outputs", exist_ok=True)  # 저장 폴더 생성(이미 있으면 무시)

image: Image.Image = result.images[0]  # result.images는 리스트 → 첫 번째 이미지 추출
# 파일명에 현재 시각(초)을 넣어 덮어쓰기 방지(매번 고유 파일명 생성)
output_path = f"./outputs/generated_{int(time.time())}.png"
image.save(output_path, format="PNG")
print(f"저장 완료: {output_path}")

# ── 이미지 정보 출력 ─────────────────────────────────────
print(f"이미지 크기: {image.size} 모드: {image.mode}")  # 예: (512, 512) / RGB
```

```python
import math

def make_image_grid(images, rows, cols):
    """생성된 이미지들을 그리드로 합성"""
    w, h = images[0].size  # 첫 이미지 크기를 기준으로 칸 크기 결정(모두 동일 크기 가정)
    # 전체 그리드용 빈 캔버스 생성(가로=cols*w, 세로=rows*h)

    for idx, img in enumerate(images):
        # idx%cols → 열 위치, idx//cols → 행 위치. 각 칸의 좌상단 좌표를 계산해 붙여넣기
        grid.paste(img, (idx % cols * w, idx // cols * h))
    return grid

images = []
for seed in [42, 137, 512, 999]:  # 시드만 바꿔 같은 프롬프트의 변형 4장을 생성

            guidance_scale=7.5,

        )
    images.append(out.images[0])  # 생성된 한 장을 리스트에 누적
```

```python
# schemas.py — Pydantic 요청/응답 스키마
# Pydantic BaseModel: 요청 본문을 자동 검증·파싱하고 Swagger 문서까지 생성해줌
from pydantic import BaseModel, Field
from typing import Optional, Literal

class ImageGenRequest(BaseModel):  # 클라이언트가 보내는 JSON 요청의 형식 정의
    prompt: str = Field(
        ...,              # ...(Ellipsis)는 필수 필드를 의미(기본값 없음)
        min_length=3,     # 너무 짧은 프롬프트 차단(최소 3자)
        max_length=500,   # 과도하게 긴 입력 차단
        description="이미지 생성에 사용할 영어 프롬프트",
        example="a serene Japanese garden at sunrise, ultra detailed, 4K"  # 문서용 예시
    )
    negative_prompt: Optional[str] = Field(
        default="blurry, low quality, watermark, deformed",  # 생략 시 사용할 기본 부정 프롬프트
        description="제외할 요소 목록"
    )
    # ge/le: 최소·최대 한계, multiple_of=64: SD가 64배수 해상도를 요구하므로 검증
    width: int = Field(default=512, ge=256, le=1024, multiple_of=64)
    height: int = Field(default=512, ge=256, le=1024, multiple_of=64)
    num_inference_steps: int = Field(default=30, ge=10, le=100)  # 역확산 스텝 수 범위 제한
    guidance_scale: float = Field(default=7.5, ge=1.0, le=20.0)  # CFG 스케일 허용 범위
    seed: Optional[int] = Field(default=None, description="재현성을 위한 랜덤 시드")  # None이면 서버가 무작위 시드 생성
    response_format: Literal["base64", "file_url"] = Field(  # 두 값 중 하나만 허용(Literal)
        default="base64",
        description="응답 형식: base64 인코딩 문자열 or 파일 URL"
    )

class ImageGenResponse(BaseModel):  # 서버가 돌려주는 응답 JSON의 형식 정의
    success: bool                       # 생성 성공 여부
    image_data: Optional[str] = None    # base64 인코딩 이미지 (response_format=base64일 때 채움)
    file_url: Optional[str] = None      # 저장된 이미지 URL (response_format=file_url일 때 채움)
    generation_time_sec: float          # 실제 생성에 걸린 시간(초)
    seed_used: int                      # 실제 사용된 시드(무작위였다면 그 값을 반환 → 재현 가능)
    prompt: str                         # 입력 프롬프트 에코
    model: str                          # 사용된 모델 ID
```

```python
# pipeline_manager.py — 모델 싱글턴 관리자
import torch
from diffusers import StableDiffusionPipeline
from threading import Lock  # 멀티스레드 환경에서 임계 구역 보호
import logging

logger = logging.getLogger(__name__)

class PipelineManager:
    """모델을 싱글턴 패턴으로 관리, 스레드 안전 추론 지원"""
    # 무거운 모델을 매 요청마다 로드하지 않도록 인스턴스를 1개만 유지(싱글턴)
    _instance = None
    _lock = Lock()            # 인스턴스 생성 시 경쟁 조건 방지
    _inference_lock = Lock()  # 동시 추론 방지 (VRAM 보호) — 한 번에 한 요청만 GPU 사용

    def __new__(cls):  # 객체 생성 시점을 가로채 싱글턴 보장
        if cls._instance is None:           # 1차 검사(락 없이 빠르게)
            with cls._lock:
                if cls._instance is None:   # 2차 검사(락 안에서 — 더블 체크 락킹)
                    cls._instance = super().__new__(cls)
                    cls._instance._pipe = None      # 아직 모델 미로드 상태
                    cls._instance._model_id = None
        return cls._instance

    def load(self, model_id: str, device: str = "cuda"):
        # 같은 모델이 이미 로드돼 있으면 재로드 생략(중복 다운로드·메모리 낭비 방지)
        if self._pipe is not None and self._model_id == model_id:
            logger.info("파이프라인이 이미 로드되어 있습니다.")
            return
        logger.info(f"모델 로딩 시작: {model_id}")
        dtype = torch.float16 if device == "cuda" else torch.float32  # GPU면 fp16, CPU면 fp32
        self._pipe = StableDiffusionPipeline.from_pretrained(
            model_id,
            torch_dtype=dtype,
            use_safetensors=True,
            safety_checker=None,
            requires_safety_checker=False
        ).to(device)
        self._pipe.set_progress_bar_config(disable=True)  # 서버 로그에 진행률 바 출력 끔
        self._model_id = model_id  # 현재 로드된 모델 ID 기록(중복 로드 판단용)
        logger.info("모델 로드 완료")

    def generate(self, **kwargs) -> list:
        """스레드 안전 이미지 생성"""
        with self._inference_lock:          # 동시 호출이 와도 한 번에 하나씩 GPU 사용
            with torch.inference_mode():    # 추론 모드(그래디언트 비활성)로 실행
                result = self._pipe(**kwargs)  # 전달된 파라미터로 실제 생성 수행
        return result.images                # PIL 이미지 리스트 반환

    @property
    def model_id(self):  # 외부에서 현재 모델 ID를 읽기 전용으로 조회
        return self._model_id

pipeline_manager = PipelineManager()  # 모듈 import 시 단일 인스턴스 생성 → 어디서든 공유
```

### (2) 입력: 프롬프트 문자열, 출력: 이미지 파일 또는 base64 인코딩

이미지 반환 방식은 크게 두 가지입니다. base64 인코딩은 JSON 응답에 이미지 데이터를 직접 포함하여 추가 요청 없이 이미지를 전달하며, `StreamingResponse`는 이미지 파일을 직접 스트리밍하여 브라우저에서 바로 표시할 수 있습니다.

| 반환 방식 | 장점 | 단점 | 권장 사용 상황 |
| --- | --- | --- | --- |
| base64 (JSON) | JSON에 포함, 단일 요청으로 완결 | 데이터 크기 33% 증가, 파싱 필요 | API 통합, 프론트엔드 처리 |
| StreamingResponse | 직접 다운로드, 효율적 전송 | 별도 HTTP 요청 구조 필요 | 브라우저 직접 표시, 다운로드 |
| 파일 URL | 대용량 이미지, CDN 연동 가능 | 스토리지 관리 필요 | 장기 보관, 대용량 이미지 |

```python
# utils.py — 이미지 변환 유틸리티
import io, base64, os, time  # io.BytesIO: 메모리 내 임시 바이트 버퍼
from PIL import Image

def pil_to_base64(image: Image.Image, format: str = "PNG") -> str:
    """PIL Image를 base64 문자열로 변환"""
    buffer = io.BytesIO()              # 파일 대신 메모리에 이미지 기록
    image.save(buffer, format=format)  # 이미지를 지정 포맷으로 버퍼에 인코딩
    buffer.seek(0)                     # 읽기 위해 버퍼 포인터를 맨 앞으로 이동
    # 바이트를 base64로 인코딩 후 ASCII 문자열로 디코딩 → JSON에 담을 수 있는 텍스트가 됨
    return base64.b64encode(buffer.read()).decode("utf-8")

def pil_to_bytes(image: Image.Image, format: str = "PNG") -> bytes:
    """PIL Image를 bytes로 변환"""
    buffer = io.BytesIO()
    image.save(buffer, format=format)
    buffer.seek(0)
    return buffer.read()  # StreamingResponse 등 바이너리 전송에 사용할 원시 바이트 반환

def save_image(image: Image.Image, output_dir: str = "./outputs") -> str:
    """이미지를 파일로 저장하고 경로 반환"""
    os.makedirs(output_dir, exist_ok=True)            # 저장 폴더 보장
    filename = f"img_{int(time.time()*1000)}.png"     # 밀리초 타임스탬프로 고유 파일명 생성
    path = os.path.join(output_dir, filename)         # OS에 맞는 경로 결합
    image.save(path)
    return path                                       # 저장 경로 반환(이후 URL 생성에 사용)

def base64_to_pil(b64_string: str) -> Image.Image:
    """base64 문자열을 PIL Image로 변환"""
    image_bytes = base64.b64decode(b64_string)        # base64 → 원시 바이트로 디코딩
```

```python
@app.post("/api/v1/generate", response_model=ImageGenResponse)  # POST 엔드포인트 + 응답 스키마 지정
async def generate_image(request: ImageGenRequest) -> ImageGenResponse:
    """텍스트 프롬프트로 이미지를 생성하고 base64 또는 URL로 반환합니다."""
    # 시드가 주어지면 그대로, 없으면 32비트 범위 내 무작위 시드 생성
    seed = request.seed if request.seed is not None else random.randint(0, 2**32 - 1)
    generator = torch.Generator("cuda" if torch.cuda.is_available() else "cpu")
    generator.manual_seed(seed)  # 시드를 고정해 결과 재현성 확보

    start = time.time()
    try:
        images = pipeline_manager.generate(  # 스레드 안전 매니저를 통해 실제 생성

            negative_prompt=request.negative_prompt,
            width=request.width,
            height=request.height,
            num_inference_steps=request.num_inference_steps,
            guidance_scale=request.guidance_scale,
            generator=generator
        )
    except RuntimeError as e:
        # GPU 메모리 부족(OOM)은 별도 처리 → 캐시 비우고 507 상태코드로 안내
        if "out of memory" in str(e).lower():

            raise HTTPException(status_code=507, detail="GPU 메모리 부족. 이미지 크기나 스텝을 줄여주세요.")
        # 그 외 런타임 오류는 500으로 전달
        raise HTTPException(status_code=500, detail=str(e))

    elapsed = time.time() - start  # 생성 소요 시간
    image = images[0]              # 첫 번째 이미지 사용

    if request.response_format == "base64":

            success=True, image_data=image_data,
            generation_time_sec=round(elapsed, 2),
            seed_used=seed, prompt=request.prompt,  # 사용된 시드 반환 → 동일 결과 재현 가능
            model=pipeline_manager.model_id
        )
    else:
        # file_url 방식: 이미지를 파일로 저장하고 접근 URL만 응답에 담음
        path = save_image(image)
        filename = os.path.basename(path)           # 경로에서 파일명만 추출
        return ImageGenResponse(
            success=True, file_url=f"/outputs/{filename}",  # 정적 마운트 경로와 결합한 URL
            generation_time_sec=round(elapsed, 2),
            seed_used=seed, prompt=request.prompt,
            model=pipeline_manager.model_id
        )
```

### (4) 이미지 생성 시간 최적화: torch.autocast / fp16

이미지 생성 속도와 메모리 효율을 개선하기 위한 주요 기법들을 소개합니다.

```python
import torch
from diffusers import StableDiffusionPipeline

# ── 방법 1: fp16 + torch.inference_mode (기본 권장) ─────
pipe = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,      # FP16: 메모리 절반, 속도 향상
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
).to("cuda")

with torch.inference_mode():  # 추론 전용 모드 → 그래디언트 추적 제거로 가장 효율적
    image = pipe("a red fox in snow", num_inference_steps=30).images[0]

# ── 방법 2: torch.autocast (구버전 호환, 동적 혼합 정밀도) ─
# 연산별로 자동으로 fp16/fp32를 섞어 사용(혼합 정밀도). 구버전 코드 호환용
with torch.autocast("cuda"):
    image = pipe("a red fox in snow", num_inference_steps=30).images[0]

# ── 방법 3: xFormers 메모리 효율적 Attention ─────────────
# pip install xformers 필요
pipe.enable_xformers_memory_efficient_attention()  # 메모리 효율 attention 커널로 교체
# VRAM 약 20~30% 절감, 속도 약 10~20% 향상

# ── 방법 4: attention slicing (저사양 GPU용) ─────────────
# VRAM이 6GB 미만인 경우 사용. 속도는 느려지지만 VRAM 절감
pipe.enable_attention_slicing(1)  # 인자 1 = attention을 한 번에 한 조각씩 계산(최대 절감)

# ── 방법 5: vae slicing (배치 처리 시 VRAM 절감) ─────────
pipe.enable_vae_slicing()  # VAE 디코딩을 이미지 단위로 쪼개 처리 → 배치 시 메모리 피크 완화

# ── 방법 6: torch.compile (PyTorch 2.0+, 첫 실행 느림) ──
# UNet을 미리 컴파일해 최적화된 커널 생성(첫 실행은 컴파일 비용으로 느림)
pipe.unet = torch.compile(pipe.unet, mode="reduce-overhead", fullgraph=True)
# 컴파일 후 반복 실행 시 20~50% 속도 향상

# ── 최적화 효과 측정 ─────────────────────────────────────
import time
times = []
for _ in range(5):  # 5회 반복 측정해 평균을 냄(첫 회 워밍업 영향 완화)
    start = time.time()
    with torch.inference_mode():
        pipe("benchmark test", num_inference_steps=20)
    times.append(time.time() - start)

print(f"평균 생성 시간: {sum(times)/len(times):.2f}초")
# memory_allocated: 현재 실제 할당된 VRAM(byte) → GB로 환산해 출력
print(f"VRAM 사용량: {torch.cuda.memory_allocated()/1e9:.2f} GB")
```

## 4.3 ControlNet과 LoRA로 이미지 변환

### (1) 이미지 구조 보존을 위한 ControlNet 개념 설명

ControlNet은 2023년 Stanford 연구팀이 발표한 기법으로, 기존 Stable Diffusion에 추가적인 조건 입력(Conditioning Input)을 제공하여 생성 이미지의 구조, 포즈, 깊이감 등을 정밀하게 제어할 수 있게 합니다. 텍스트 프롬프트만으로는 세밀한 구도와 구조를 원하는 대로 제어하기 어렵다는 SD의 한계를 극복하는 핵심 기술입니다.

ControlNet은 원본 UNet의 가중치를 동결(freeze)하고, 별도의 학습 가능한 복사본(Trainable Copy)을 통해 조건 신호를 주입합니다. 이를 통해 원본 모델의 이미지 품질을 유지하면서 구조 정보를 정확하게 반영할 수 있습니다.

| ControlNet 유형 | 입력 조건 | 모델 ID | 활용 예 |
| --- | --- | --- | --- |
| Canny Edge | 캐니 엣지 검출 이미지 | `lllyasviel/sd-controlnet-canny` | 선화 → 채색, 스케치 변환 |
| HED Edge | 부드러운 엣지 맵 | `lllyasviel/sd-controlnet-hed` | 윤곽 보존 스타일 변환 |
| OpenPose | 인체 관절 스켈레톤 | `lllyasviel/sd-controlnet-openpose` | 자세 유지 캐릭터 생성 |
| Depth Map | 깊이 정보 맵 | `lllyasviel/sd-controlnet-depth` | 3D 구도 보존 재구성 |
| Normal Map | 법선 벡터 맵 | `lllyasviel/sd-controlnet-normal` | 3D 표면 재질 변환 |
| Segmentation | 의미적 분할 맵 | `lllyasviel/sd-controlnet-seg` | 배경/객체 구조 보존 |
| Scribble | 손그림 낙서 맵 | `lllyasviel/sd-controlnet-scribble` | 러프 스케치 → 완성 이미지 |
| MLSD Line | 직선 구조 검출 | `lllyasviel/sd-controlnet-mlsd` | 건축물, 실내 구조 보존 |

ControlNet의 작동 방식을 단계별로 살펴보면 다음과 같습니다.

#### ControlNet 처리 흐름

1. **조건 이미지 전처리**: 원본 이미지 → 조건 맵 생성
   - 예) 원본 사진 → Canny 엣지 검출 → 흑백 엣지 이미지
2. **ControlNet Encoder 처리**: 조건 맵 → 중간 Feature Map 시퀀스 생성
   - (원본 UNet Encoder와 동일한 구조, 별도 학습 가중치)
3. **Feature 주입**: ControlNet의 출력이 원본 UNet의 각 디코더 블록에 더해짐
4. **통상적 역확산 진행**: 텍스트 임베딩 + ControlNet 조건이 결합된 UNet으로 노이즈 제거 반복 → 최종 이미지 생성

### (2) ControlNet Canny 예시 모델 실습

가장 널리 사용되는 ControlNet Canny를 이용하여 입력 이미지의 윤곽선 구조를 보존하면서 새로운 스타일의 이미지를 생성합니다.

```python
import torch
import cv2          # OpenCV: Canny 엣지 검출 등 이미지 전처리
import numpy as np
from PIL import Image
from diffusers import StableDiffusionControlNetPipeline, ControlNetModel
from diffusers import UniPCMultistepScheduler  # 적은 스텝으로 고품질 내는 스케줄러

# ── Step 1: ControlNet 모델 로드 ─────────────────────────
# Canny 엣지 조건을 처리하는 보조 신경망. SD 본체와 별개로 로드해 결합한다
controlnet = ControlNetModel.from_pretrained(
    "lllyasviel/sd-controlnet-canny",
    torch_dtype=torch.float16,
    use_safetensors=True
).to("cuda")

# ── Step 2: 파이프라인 조합 ──────────────────────────────
# SD 본체에 controlnet을 끼워 구조 제어가 가능한 파이프라인 구성
pipe = StableDiffusionControlNetPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    controlnet=controlnet,   # 위에서 로드한 ControlNet 주입
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
).to("cuda")

# UniPCMultistepScheduler: 고품질 + 빠른 생성 (20 스텝으로 충분)
# 기존 스케줄러 설정을 그대로 가져와 UniPC로 교체 → 적은 스텝으로 좋은 품질
pipe.scheduler = UniPCMultistepScheduler.from_config(pipe.scheduler.config)
pipe.enable_xformers_memory_efficient_attention()  # VRAM 절감(ControlNet은 메모리를 더 씀)
```

```python
# ── Step 3: 입력 이미지에서 Canny 엣지 추출 ─────────────
def extract_canny_edges(image_path: str, low_threshold=100, high_threshold=200) -> Image.Image:
    """이미지에서 Canny 엣지를 검출하여 ControlNet 입력 이미지 생성"""
    # 이미지 로드 및 리사이즈 — SD 1.5 기본 해상도인 512x512로 맞춤
    img = Image.open(image_path).convert("RGB").resize((512, 512))
    img_array = np.array(img)  # PIL → NumPy 배열(OpenCV가 다루는 형식)

    # BGR로 변환 후 Canny 엣지 검출 (OpenCV는 BGR 순서를 기본 사용)
    img_bgr = cv2.cvtColor(img_array, cv2.COLOR_RGB2BGR)
    # Canny: 두 임계값 사이 변화량을 윤곽선으로 검출(낮을수록 더 많은 엣지)
    edges = cv2.Canny(img_bgr, low_threshold, high_threshold)

    # 3채널로 변환 (ControlNet 입력 형식) — 엣지는 1채널 흑백이라 RGB 3채널로 확장
    edges_rgb = cv2.cvtColor(edges, cv2.COLOR_GRAY2RGB)
    return Image.fromarray(edges_rgb)  # NumPy → PIL 이미지로 되돌려 반환

# ── Step 4: ControlNet으로 이미지 생성 ──────────────────
def generate_with_canny(
    image_path: str,
    prompt: str,
    negative_prompt: str = "blurry, low quality, deformed",
    num_steps: int = 20,
    guidance_scale: float = 7.5,
    controlnet_scale: float = 1.0,  # ControlNet 조건 강도 (0.5~1.5). 높을수록 원본 구조 강하게 보존
    seed: int = 42
) -> tuple[Image.Image, Image.Image]:
    """원본 이미지 엣지를 보존하며 새로운 스타일로 이미지 생성"""

    # Canny 엣지 맵 생성 (구조 조건으로 사용)
    canny_image = extract_canny_edges(image_path)

    generator = torch.Generator("cuda").manual_seed(seed)  # 결과 재현용 시드 고정

    with torch.inference_mode():
        result = pipe(
            prompt=prompt,
            negative_prompt=negative_prompt,
            image=canny_image,            # ControlNet 조건 입력(엣지 맵으로 구도를 안내)
            num_inference_steps=num_steps,
            guidance_scale=guidance_scale,
            controlnet_conditioning_scale=controlnet_scale,  # 엣지 조건을 얼마나 강하게 반영할지
            generator=generator
        )

    # (조건으로 쓴 엣지 맵, 최종 생성 이미지) 두 가지를 함께 반환
    return canny_image, result.images[0]

# ── 실행 테스트 ──────────────────────────────────────────
canny_map, generated = generate_with_canny(
    image_path="./sample_building.jpg",
    prompt="a futuristic cyberpunk building, neon lights, rain, ultra detailed, 8K",
    controlnet_scale=0.9  # 0.9 → 원본 건물 구조를 비교적 강하게 유지
)

canny_map.save("./outputs/canny_map.png")        # 입력으로 쓴 엣지 맵 저장(확인용)
generated.save("./outputs/canny_result.png")     # 변환 결과 저장
print("ControlNet Canny 결과 저장 완료")
```

### (3) Low-Rank Adaptation (LoRA)란 무엇인가?

LoRA(Low-Rank Adaptation)는 2021년 Microsoft Research에서 제안한 파라미터 효율적 파인튜닝(PEFT) 기법으로, 원본 모델의 대부분 가중치를 동결(freeze)하고 작은 행렬 두 개(저랭크 행렬 A와 B)만 학습합니다. Stable Diffusion에 적용하면 특정 아티스트 스타일, 캐릭터, 개념 등을 수십~수백 MB 크기의 소형 파일로 학습하고 배포할 수 있습니다.

#### LoRA의 수학적 원리

기존 파인튜닝: `W_new = W_original + ΔW` (`ΔW`는 `W`와 같은 크기)

LoRA: `W_new = W_original + α × (A × B)`

- `W_original`: 기존 모델 가중치 (고정, 학습 안 함)
- `A`: 저랭크 행렬 (d × r) — 학습 대상
- `B`: 저랭크 행렬 (r × k) — 학습 대상
- `r`: 랭크 (보통 4~64. 작을수록 학습 파라미터 적음)
- `α`: 스케일링 인자

효과: `r=4` 사용 시 학습 파라미터 수가 기존의 약 0.01~0.1% 수준으로 감소 → 학습 시간 단축, 소형 파일 배포, 여러 LoRA 혼합 사용 가능

| 구분 | 일반 파인튜닝 | LoRA 파인튜닝 |
| --- | --- | --- |
| 학습 파라미터 수 | 수억 개 (전체) | 수백만 개 (0.1% 수준) |
| 파일 크기 | 2~7 GB | 1~200 MB |
| 학습 시간 | 수 시간 ~ 수 일 | 수십 분 ~ 수 시간 |
| VRAM 요구 | 24 GB+ | 6~12 GB |
| 여러 개 혼합 | 불가능 | 가중치로 혼합 가능 |
| 원본 모델 호환 | 재배포 필요 | 원본 모델 + LoRA 파일 |

### (4) LoRA 가중치 로드 및 스타일 전환 실습

Hugging Face Hub 또는 civitai.com에서 다양한 LoRA 파일을 다운로드하여 기존 Stable Diffusion 모델에 적용할 수 있습니다. diffusers 라이브러리는 `load_lora_weights()` 메서드로 간편하게 LoRA를 로드합니다.

```python
import torch
from diffusers import StableDiffusionPipeline

# ── 기본 SD 파이프라인 로드 ──────────────────────────────
pipe = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
).to("cuda")

# ── LoRA 가중치 로드 방법 1: Hugging Face Hub에서 직접 로드 ─
# load_lora_weights: 원본 가중치는 그대로 두고 소형 LoRA 가중치만 얹음
# 예: 애니메이션 스타일 LoRA
pipe.load_lora_weights(
    "sayakpaul/sd-model-finetuned-lora-t4",   # Hub 저장소 ID
    weight_name="pytorch_lora_weights.safetensors"  # 저장소 내 LoRA 파일명
)

# ── LoRA 가중치 로드 방법 2: 로컬 .safetensors 파일 로드 ─
pipe.load_lora_weights(
    "./lora_weights/",          # 디렉토리 경로(직접 받은 LoRA 파일이 있는 폴더)
    weight_name="anime_style.safetensors"
)

# ── LoRA 스케일 조정 ─────────────────────────────────────
# cross_attention_kwargs={"scale": 0.7}로 LoRA 적용 강도 조절
with torch.inference_mode():
    image_lora = pipe(
        "a beautiful anime girl in a garden, cherry blossoms, soft lighting",
        negative_prompt="realistic, photo, 3D render, deformed",
        num_inference_steps=30,
        guidance_scale=7.5,
        cross_attention_kwargs={"scale": 0.8},  # LoRA 강도: 0(미적용)~1(완전 적용)
    ).images[0]

image_lora.save("./outputs/lora_result.png")

# ── 여러 LoRA 혼합 사용 ──────────────────────────────────
# (diffusers 0.26+에서 지원)
# adapter_name으로 각 LoRA에 이름표를 달아 두면 나중에 가중치를 따로 조절 가능
pipe.load_lora_weights("./lora_weights/", weight_name="style_A.safetensors",
                       adapter_name="style_A")
pipe.load_lora_weights("./lora_weights/", weight_name="character_B.safetensors",
                       adapter_name="character_B")

# 두 LoRA를 6:4 비율로 혼합 — set_adapters로 활성 LoRA와 비중을 동시에 지정
pipe.set_adapters(["style_A", "character_B"], adapter_weights=[0.6, 0.4])

# 혼합 LoRA로 생성
with torch.inference_mode():
    image_mixed = pipe(
        "a warrior princess in a fantasy world, dramatic lighting",
        num_inference_steps=30
    ).images[0]
image_mixed.save("./outputs/mixed_lora.png")

# ── LoRA 비활성화 (원본 모델로 복원) ─────────────────────
pipe.unload_lora_weights()  # 얹었던 LoRA를 제거 → 순수 SD 모델 상태로 되돌림

# ── LoRA를 모델에 영구 병합 (배포 시 권장) ───────────────
# 병합 후에는 원본 SD 파이프라인처럼 사용 가능 (LoRA 파일 불필요)
pipe.fuse_lora(lora_scale=1.0)        # LoRA 가중치를 본 모델에 직접 합침(추론 시 오버헤드 제거)
pipe.save_pretrained("./merged_model/")  # 병합된 완성 모델을 통째로 저장 → 배포용
```

### (5) 업로드된 이미지 → 변환된 이미지 반환 API 구현

사용자가 이미지를 업로드하면 ControlNet 또는 img2img 파이프라인으로 변환하여 반환하는 REST API를 구현합니다. FastAPI의 multipart/form-data 처리를 통해 이미지 파일을 수신합니다.

```python
# image_transform_router.py — 이미지 변환 라우터
import io, torch
# APIRouter: 라우트를 모듈로 분리 / UploadFile·File·Form: 파일+폼 데이터 수신용
from fastapi import APIRouter, UploadFile, File, Form, HTTPException
from fastapi.responses import StreamingResponse
from PIL import Image
from diffusers import (
    StableDiffusionImg2ImgPipeline,        # 이미지+프롬프트 → 변형 이미지
    StableDiffusionControlNetPipeline,     # 구조 보존 변환
    ControlNetModel,
    UniPCMultistepScheduler
)
from diffusers.utils import load_image
import numpy as np, cv2

# prefix: 이 라우터의 모든 경로 앞에 붙는 공통 접두어 / tags: Swagger 그룹 라벨
router = APIRouter(prefix="/api/v1/transform", tags=["Image Transform"])

# ── 파이프라인 초기화 (앱 시작 시 한 번만 실행) ──────────
# 모듈 전역에 캐시해 두고 최초 호출 때만 로드(지연 초기화 = lazy init)
_img2img_pipe = None
_controlnet_pipe = None

def get_img2img_pipe():
    global _img2img_pipe
    if _img2img_pipe is None:  # 아직 로드 안 됐으면 한 번만 로드
        _img2img_pipe = StableDiffusionImg2ImgPipeline.from_pretrained(
            "runwayml/stable-diffusion-v1-5",
            torch_dtype=torch.float16,
            use_safetensors=True,
            safety_checker=None,
            requires_safety_checker=False
        ).to("cuda")
    return _img2img_pipe  # 캐시된 파이프라인 반환

def get_controlnet_pipe():
    global _controlnet_pipe
    if _controlnet_pipe is None:
        cn = ControlNetModel.from_pretrained(   # Canny ControlNet 로드
            "lllyasviel/sd-controlnet-canny",
            torch_dtype=torch.float16, use_safetensors=True
        ).to("cuda")
        _controlnet_pipe = StableDiffusionControlNetPipeline.from_pretrained(
            "runwayml/stable-diffusion-v1-5",
            controlnet=cn,   # 위에서 만든 ControlNet 결합
            torch_dtype=torch.float16,
            use_safetensors=True,
            safety_checker=None,
            requires_safety_checker=False
        ).to("cuda")
        # 스케줄러를 UniPC로 교체 → 적은 스텝으로 고품질
        _controlnet_pipe.scheduler = UniPCMultistepScheduler.from_config(
            _controlnet_pipe.scheduler.config)
    return _controlnet_pipe
```

> **[코드 4-17]** 이미지 변환 라우터 및 파이프라인 초기화

```python
# ── img2img 엔드포인트 ───────────────────────────────────
@router.post("/img2img", summary="이미지 스타일 변환 (img2img)")
async def img2img_transform(
    # File(...): 업로드 파일 필수 / Form(...): multipart 폼 필드로 텍스트 값 수신
    file: UploadFile = File(..., description="변환할 원본 이미지 파일 (JPG/PNG)"),
    prompt: str = Form(..., description="변환 스타일 프롬프트"),
    negative_prompt: str = Form(default="blurry, low quality"),
    strength: float = Form(default=0.75, ge=0.1, le=1.0,
                           description="변형 강도. 1.0=완전 변형, 0.1=원본 유지"),
    num_steps: int = Form(default=30, ge=10, le=80),
    guidance_scale: float = Form(default=7.5)
):
    # 이미지 읽기 및 전처리
    content = await file.read()  # 업로드 파일 바이트를 비동기로 읽음
    if not file.content_type.startswith("image/"):  # MIME 타입 검증(이미지가 아니면 거부)
        raise HTTPException(400, "이미지 파일만 업로드 가능합니다.")

    # 바이트 → PIL 이미지로 열고 RGB·512px로 정규화(파이프라인 입력 규격 맞춤)
    init_image = Image.open(io.BytesIO(content)).convert("RGB").resize((512, 512))
    pipe = get_img2img_pipe()  # 캐시된 img2img 파이프라인 획득

    with torch.inference_mode():
        result = pipe(
            prompt=prompt,
            negative_prompt=negative_prompt,
            image=init_image,           # 변형의 출발점이 되는 원본 이미지
            strength=strength,          # 변형 강도(낮으면 원본 유지, 높으면 크게 변형)
            num_inference_steps=num_steps,
            guidance_scale=guidance_scale,
        )

    output_image = result.images[0]
    buf = io.BytesIO()                  # 파일 저장 없이 메모리 버퍼에 바로 인코딩
    output_image.save(buf, format="PNG")
    buf.seek(0)                         # 스트리밍 위해 버퍼를 처음으로 되감기
    # StreamingResponse: 이미지 바이트를 그대로 응답 본문으로 전송(브라우저에서 바로 표시)
    return StreamingResponse(buf, media_type="image/png",
        headers={"Content-Disposition": "inline; filename=transformed.png"})

# ── ControlNet 변환 엔드포인트 ───────────────────────────
@router.post("/controlnet-canny", summary="ControlNet Canny 구조 보존 변환")
async def controlnet_canny_transform(
    file: UploadFile = File(...),
    prompt: str = Form(...),
    negative_prompt: str = Form(default="blurry, deformed, low quality"),
    num_steps: int = Form(default=20),
    controlnet_scale: float = Form(default=1.0, ge=0.1, le=2.0),  # 구조 보존 강도
):
    content = await file.read()
    orig_image = Image.open(io.BytesIO(content)).convert("RGB").resize((512, 512))

    # Canny 엣지 추출 — 업로드 이미지의 윤곽선을 조건 맵으로 변환
    arr = np.array(orig_image)
    bgr = cv2.cvtColor(arr, cv2.COLOR_RGB2BGR)              # OpenCV용 BGR로 변환
    edges = cv2.Canny(bgr, 100, 200)                       # 엣지 검출(임계값 100~200)
    canny_image = Image.fromarray(cv2.cvtColor(edges, cv2.COLOR_GRAY2RGB))  # 3채널 PIL로 변환

    pipe = get_controlnet_pipe()
    with torch.inference_mode():
        result = pipe(
            prompt=prompt,
            negative_prompt=negative_prompt,
            image=canny_image,                       # 엣지 맵을 구조 조건으로 입력
            num_inference_steps=num_steps,
            controlnet_conditioning_scale=controlnet_scale,  # 엣지 조건 반영 강도
        )

    buf = io.BytesIO()
    result.images[0].save(buf, format="PNG")
    buf.seek(0)
    return StreamingResponse(buf, media_type="image/png")  # PNG 바이트 스트리밍 응답
```

> **[코드 4-18]** img2img 및 ControlNet Canny 변환 엔드포인트 전체 코드

> `strength` 파라미터는 img2img 파이프라인에서 원본 이미지를 얼마나 변형할지 결정합니다. 0.3~0.5이면 원본 이미지와 유사하면서 스타일만 변경되고, 0.7~0.9이면 구도는 비슷하지만 크게 변형됩니다.

## 4.4 서버에서 모델 최적화하기

### (1) GPU 메모리 최적화

Stable Diffusion 모델은 기본적으로 상당한 GPU 메모리(VRAM)를 필요로 합니다. SD v1.5를 fp32로 로드하면 약 8GB, fp16으로는 약 4GB의 VRAM을 사용합니다. VRAM이 제한된 환경에서는 다양한 최적화 기법을 조합하여 메모리 사용량을 줄여야 합니다.

| VRAM 용량 | 가능한 모델 / 설정 | 권장 최적화 조합 |
| --- | --- | --- |
| 4 GB | SD v1.5 (fp16, 512px) | fp16 + attention_slicing + vae_slicing |
| 6 GB | SD v1.5 (fp16, 512px), SD v2 (fp16, 512px) | fp16 + xFormers + attention_slicing |
| 8 GB | SD v1.5/v2 (fp16, 768px), ControlNet (512px) | fp16 + xFormers |
| 12 GB | SD v1.5/v2 (fp16, 768px), ControlNet (768px) | fp16 + xFormers (선택) |
| 16 GB+ | SDXL (fp16, 1024px), 배치 생성 가능 | fp16 (필수), 나머지 선택적 |
| 24 GB+ | SDXL + ControlNet, 대형 배치 | 최적화 없이도 쾌적한 실행 |

```python
import torch
from diffusers import StableDiffusionPipeline

# ── 현재 VRAM 사용량 모니터링 함수 ──────────────────────
def print_gpu_memory(label: str = ""):
    if torch.cuda.is_available():
        allocated = torch.cuda.memory_allocated() / 1e9   # 실제 텐서가 점유한 VRAM
        reserved = torch.cuda.memory_reserved() / 1e9     # 캐시 포함 PyTorch가 예약한 VRAM
        total = torch.cuda.get_device_properties(0).total_memory / 1e9  # GPU 전체 용량
        print(f"[{label}] VRAM: 사용={allocated:.2f}GB / 예약={reserved:.2f}GB / 전체={total:.2f}GB")

print_gpu_memory("모델 로드 전")  # 기준점(거의 0) 확인

# ── 기본 메모리 최적화 파이프라인 초기화 ─────────────────
pipe = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
).to("cuda")

print_gpu_memory("기본 로드 후")  # 모델이 차지한 VRAM 확인(fp16 기준 약 4GB)

# ── xFormers 활성화 (VRAM 20~30% 절감) ──────────────────
try:
    pipe.enable_xformers_memory_efficient_attention()  # 설치돼 있으면 활성화
    print("xFormers 활성화 성공")
except Exception:
    # xFormers가 없으면 예외 발생 → attention slicing으로 대체(둘 다 메모리 절감 목적)
    print("xFormers 미설치 — attention_slicing으로 대체")
    pipe.enable_attention_slicing(1)

print_gpu_memory("최적화 적용 후")  # 최적화 전후 VRAM 변화 비교

# ── 사용하지 않는 구성 요소 언로드 ──────────────────────
# 텍스트 인코더는 생성 전에 한 번만 필요 → 사용 후 오프로드 가능
def generate_and_offload(pipe, prompt, **kwargs):
    """생성 후 텍스트 인코더를 CPU로 이동하여 VRAM 확보"""
    image = pipe(prompt, **kwargs).images[0]
    # 텍스트 인코더를 CPU로 이동 → 그만큼 VRAM 반환
    pipe.text_encoder.to("cpu")
    torch.cuda.empty_cache()  # PyTorch가 잡고 있던 캐시까지 OS에 반환
    print_gpu_memory("생성 후 오프로드")
    # 다음 생성 전에 GPU로 복귀 필요(텍스트 인코딩은 GPU에서 해야 빠름)
    pipe.text_encoder.to("cuda")
    return image
```

### (2) enable_model_cpu_offload()로 메모리 절약

`enable_model_cpu_offload()`는 diffusers가 제공하는 강력한 메모리 최적화 기법입니다. 모델의 각 구성 요소(Text Encoder, UNet, VAE)를 필요할 때만 GPU에 올리고, 사용 후에는 자동으로 CPU로 이동하여 VRAM 사용량을 최소화합니다.

```python
import torch
from diffusers import StableDiffusionPipeline, StableDiffusionXLPipeline

# ── CPU Offload 방법 1: enable_model_cpu_offload ─────────
# 모델 구성 요소를 자동으로 CPU ↔ GPU 이동
# accelerate 패키지 필요: pip install accelerate
pipe = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
)  # ← .to("cuda") 하지 않음! (오프로드가 장치 이동을 직접 관리하므로)

# CPU Offload 활성화 (4GB GPU에서도 SD 실행 가능)
# 구성 요소(텍스트인코더/UNet/VAE)를 쓸 때만 GPU로 올리고 끝나면 CPU로 내림
pipe.enable_model_cpu_offload()

# 이후 사용법은 동일
with torch.inference_mode():
    image = pipe("a sunset over the ocean, impressionist painting style").images[0]
image.save("./outputs/cpu_offload_test.png")

# ── CPU Offload 방법 2: enable_sequential_cpu_offload ────
# 더 세밀한 레이어 단위 오프로드. VRAM 최소화하지만 느림
pipe2 = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
)
pipe2.enable_sequential_cpu_offload()  # 레이어 단위로 순차 이동 → VRAM 최소(가장 느림)

# ── SDXL에서의 CPU Offload ─────────────────────────────
# SDXL은 10GB+ VRAM 필요 → CPU Offload로 8GB GPU에서 실행 가능
pipe_xl = StableDiffusionXLPipeline.from_pretrained(
    "stabilityai/stable-diffusion-xl-base-1.0",
    torch_dtype=torch.float16,
    use_safetensors=True,
    variant="fp16"
)
pipe_xl.enable_model_cpu_offload()  # SDXL도 동일하게 오프로드로 저사양 GPU 대응

with torch.inference_mode():
    image_xl = pipe_xl(
        "a breathtaking landscape, masterpiece quality, 8K",
        num_inference_steps=25,
    ).images[0]
image_xl.save("./outputs/sdxl_cpu_offload.png")
```

> **[코드 4-20]** enable_model_cpu_offload() 및 sequential_cpu_offload 사용법

| Offload 방법 | VRAM 사용량 | 속도 | 설명 |
| --- | --- | --- | --- |
| `.to("cuda")` (기본) | 전체 (~4GB fp16) | 가장 빠름 | 모든 구성 요소가 GPU에 상주 |
| `enable_model_cpu_offload()` | 최소 ~2.5GB | 약간 느림 | 사용하지 않는 구성 요소는 CPU로 |
| `enable_sequential_cpu_offload()` | 최소 ~1.5GB | 느림 | 레이어 단위 순차 오프로드 |

### (3) 실시간 생성 속도 개선 전략

서비스 환경에서 사용자 대기 시간을 최소화하기 위한 다양한 전략을 소개합니다.

```python
import torch
from diffusers import StableDiffusionPipeline, LCMScheduler

# ── 전략 1: LCM (Latent Consistency Model) 스케줄러 ──────
# 4~8 스텝으로 고품질 이미지 생성 가능! 기존 30~50 스텝 대비 5~10배 빠름
# pip install diffusers[torch] --upgrade 필요

pipe_lcm = StableDiffusionPipeline.from_pretrained(
    "runwayml/stable-diffusion-v1-5",
    torch_dtype=torch.float16,
    use_safetensors=True,
    safety_checker=None,
    requires_safety_checker=False
).to("cuda")

# LCM-LoRA 로드 (LCM 속도를 일반 SD 모델에 적용) — 모델 교체 없이 LoRA만 얹어 가속
pipe_lcm.load_lora_weights("latent-consistency/lcm-lora-sdv1-5")
# 스케줄러를 LCM 전용으로 교체해야 소수 스텝 생성이 정상 동작
pipe_lcm.scheduler = LCMScheduler.from_config(pipe_lcm.scheduler.config)

# 4 스텝으로 빠른 생성
import time
start = time.time()
with torch.inference_mode():
    image_lcm = pipe_lcm(
        "a beautiful forest waterfall, golden light, photorealistic",
        num_inference_steps=4,        # 기존 30~50 스텝 대신 4 스텝!
        guidance_scale=1.0,           # LCM은 CFG가 높으면 오히려 망가짐 → 1.0 권장
        cross_attention_kwargs={"scale": 1.0}  # LCM-LoRA 적용 강도
    ).images[0]
print(f"LCM 4-step 생성 시간: {time.time()-start:.2f}초")
image_lcm.save("./outputs/lcm_4step.png")

# ── 전략 2: Tiny AutoEncoder (TAESD) ─────────────────────
# VAE 디코딩 단계를 초경량 TAESD로 대체 → 최종 디코딩 3~5배 빠름
# 품질 소폭 저하 있으나 실시간 프리뷰에 적합
from diffusers import AutoencoderTiny

# 파이프라인의 VAE를 경량 TAESD로 교체(잠재→픽셀 변환만 담당하는 부분 가속)
pipe.vae = AutoencoderTiny.from_pretrained(
    "madebyollin/taesd",
    torch_dtype=torch.float16
).to("cuda")

# ── 전략 3: 워밍업 실행으로 첫 응답 지연 제거 ───────────
# torch.compile 사용 시 첫 실행이 매우 느림 → 워밍업으로 해결
print("모델 워밍업 중...")
with torch.inference_mode():
    _ = pipe("warmup", num_inference_steps=1)  # 1 스텝 더미 실행으로 컴파일·캐시 미리 준비
torch.cuda.empty_cache()
print("워밍업 완료. 이제 빠른 응답 가능.")

# ── 전략 4: 배치 생성 파이프라인 (처리량 최대화) ─────────
def batch_generate(prompts: list, **kwargs) -> list:
    """여러 프롬프트를 한 번의 forward pass로 처리"""
    with torch.inference_mode():
        results = pipe(prompts, **kwargs)  # 프롬프트 리스트를 넘기면 한 번에 배치 처리
    return results.images

prompts = [  # 서로 다른 3개의 프롬프트를 한 배치로 묶음
    "a red sports car on a mountain road",
    "a blue sailboat on a calm lake",
    "a golden wheat field at dusk"
]

start = time.time()
images_batch = batch_generate(prompts, num_inference_steps=20)
elapsed = time.time() - start
# 총 시간과 장당 평균 시간 비교 → 배치가 개별 생성보다 처리량이 높음을 확인
print(f"배치(3장) 생성 시간: {elapsed:.2f}초 (장당 {elapsed/3:.2f}초)")
```

## 연습 문제

1. `StableDiffusionPipeline`을 로드하고 다음 프롬프트들로 각각 이미지를 생성한 뒤, `num_inference_steps`를 20, 35, 50으로 달리하여 생성된 이미지의 품질 차이를 비교하고 결과를 정리하세요. 프롬프트: `"a majestic eagle soaring above snowy mountains, golden light, 4K"`
2. 4.2절의 FastAPI 이미지 생성 API 서버를 로컬에 구동하고, Swagger UI(`/docs`)에서 직접 프롬프트를 입력하여 이미지를 생성해 보세요. 생성된 이미지를 base64 형식으로 받아 PNG 파일로 저장하는 Python 클라이언트 스크립트를 작성하세요.
3. ControlNet Canny 파이프라인을 사용하여 임의의 건물 또는 인물 사진을 입력으로, 다음의 두 프롬프트로 각각 이미지를 생성하고 비교하세요. (A) `"a futuristic cyberpunk building, neon lights"` (B) `"a traditional Japanese architecture, cherry blossoms"`. `controlnet_conditioning_scale`을 0.5와 1.0으로 각각 적용하여 결과 차이도 관찰하세요.
4. `enable_model_cpu_offload()`를 적용한 파이프라인과 적용하지 않은 파이프라인의 (A) VRAM 사용량과 (B) 이미지 생성 시간을 측정하여 비교 분석하는 코드를 작성하세요. GPU 환경이 없다면 `torch.cuda.memory_allocated()` 부분을 제외하고 시간만 비교해도 됩니다.
5. (심화) Hugging Face Spaces에 무료 계정으로 Space를 만들고, 4.4절의 Gradio 앱 코드(app.py)를 업로드하여 실제로 배포해 보세요. 배포된 Space URL을 공유할 수 있게 구성하고, 배포 과정에서 발생한 오류와 해결 방법을 정리하세요.
