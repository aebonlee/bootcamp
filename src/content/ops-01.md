# Git & GitHub 입문 가이드

버전 관리가 처음이어도 괜찮습니다. 이 가이드는 "버전 관리가 왜 필요한가"라는 근본 개념에서 출발해, Git의 동작 원리, 브랜치와 병합, GitHub를 이용한 협업, 좋은 커밋 습관과 오픈소스 라이선스, 그리고 실수했을 때 살아남는 고급 기능까지 한 번에 다룹니다. 명령어를 외우기보다 "지금 무슨 일이 일어나는지"를 이해하는 데 초점을 맞췄으니, 따라 치면서 손에 익히면 됩니다.

---

## 1.1 버전 관리란?

버전 관리(Version Control)는 **파일의 변경 이력을 시간순으로 기록하고 추적하는 시스템**입니다. `보고서_최종.docx`, `보고서_최종_진짜최종.docx`, `보고서_최종_진짜최종_v2.docx` 처럼 파일을 복사해서 이름을 바꿔가며 관리하던 혼란을 끝내 줍니다.

### 왜 필요한가

파일을 직접 복사해 백업하는 방식에는 한계가 분명합니다. 어떤 버전이 최신인지 헷갈리고, 무엇을 왜 바꿨는지 기록이 남지 않으며, 여러 사람이 동시에 같은 파일을 고치면 누군가의 작업이 통째로 사라집니다. 버전 관리 시스템은 이 모든 문제를 구조적으로 해결합니다.

버전 관리가 주는 세 가지 핵심 가치는 다음과 같습니다.

| 가치 | 설명 | 실제 효과 |
| --- | --- | --- |
| 이력 추적 | 모든 변경을 **스냅샷**으로 저장해 언제든 비교·복원 | "3일 전 잘 되던 코드"로 1초 만에 되돌아가기 |
| 협업 | 여러 사람이 충돌 없이 동시에 작업 | 각자 작업하고 안전하게 합치기 |
| 안전망 | 마음껏 실험하고 잘못되면 되돌리기 | "망가뜨리면 어쩌지" 하는 두려움 제거 |

### 핵심 개념: 스냅샷

Git은 변경을 저장할 때, 바뀐 줄만 따로 기록하는 것이 아니라 그 시점의 프로젝트 전체 상태를 **스냅샷(snapshot)** 으로 찍어 둡니다. 사진을 찍듯 "이 순간의 모습"을 기록해 두기 때문에, 나중에 어떤 시점으로든 정확하게 되돌아갈 수 있습니다. 각 스냅샷에는 누가, 언제, 무엇을, 왜 바꿨는지가 함께 남습니다.

### 초보자가 흔히 하는 오해

- "버전 관리는 큰 회사 프로젝트에서나 쓰는 것" — 아닙니다. 혼자 하는 작은 프로젝트에서도 "어제 잘 되던 코드로 돌아가기" 기능 하나만으로도 충분히 값집니다.
- "백업이랑 똑같은 것 아닌가?" — 백업은 단순 복사본이지만, 버전 관리는 변경의 **이유와 맥락**까지 기록하고 특정 부분만 골라 되돌리는 정교한 작업이 가능합니다.

## 1.2 Git vs GitHub

입문자가 가장 많이 헷갈리는 부분입니다. **Git과 GitHub는 다른 것**입니다. 한쪽은 도구(Tool)이고, 다른 한쪽은 그 도구를 위한 플랫폼(Platform)입니다.

| 구분 | Git | GitHub |
| --- | --- | --- |
| 성격 | TOOL (도구) | PLATFORM (플랫폼·서비스) |
| 위치 | 내 컴퓨터(로컬)에서 동작 | 클라우드(인터넷)에서 동작 |
| 역할 | 커밋·브랜치·병합의 핵심 **엔진** | Git 저장소를 **호스팅**하고 협업 지원 |
| 인터넷 | 없어도 동작 | 접속 필요 |
| 대표 기능 | 변경 기록, 이력 관리 | 협업, 코드 리뷰, 이슈(Issue), PR(Pull Request) |

### 비유로 이해하기

- **Git** 은 글을 쓰고 고치는 "워드프로세서 + 변경 기록 기능"이라고 생각하면 됩니다. 내 컴퓨터 안에서 혼자서도 완벽하게 동작합니다.
- **GitHub** 는 그 문서를 올려 두고 다른 사람과 함께 보고, 의견을 달고, 같이 고치는 "클라우드 협업 공간"입니다. GitHub 외에도 GitLab, Bitbucket 같은 비슷한 플랫폼이 있습니다.

즉, **Git이 먼저이고 GitHub는 선택**입니다. Git만으로도 버전 관리는 완전하게 됩니다. 다만 다른 사람과 협업하거나, 코드를 안전하게 클라우드에 보관하고 싶을 때 GitHub를 더하는 것입니다.

## 1.3 Git의 3가지 영역과 첫 커밋

Git을 제대로 이해하려면, 변경 사항이 흘러가는 **3개의 영역**을 알아야 합니다. 이 흐름을 이해하면 `git add`와 `git commit`이 왜 두 단계로 나뉘어 있는지 자연스럽게 납득됩니다.

```text
작업 디렉터리             스테이징 영역              저장소
Working Directory  ──→   Staging Area      ──→    Repository
                 git add                git commit
```

| 영역 | 영문 | 의미 | 비유 |
| --- | --- | --- | --- |
| 작업 디렉터리 | Working Directory | 지금 내가 실제로 파일을 편집하는 공간 | 작업 책상 위 |
| 스테이징 영역 | Staging Area (Index) | 다음 커밋에 포함할 변경을 모아 두는 대기실 | 택배 상자에 담기 |
| 저장소 | Repository | 커밋이 영구히 기록되는 공간 | 창고에 보관 |

### 왜 스테이징이라는 중간 단계가 있을까

"수정하면 바로 저장되지 왜 두 번 거치나?" 싶지만, 이 중간 단계 덕분에 **변경 중 일부만 골라서 커밋**할 수 있습니다. 예를 들어 한 번에 여러 파일을 고쳤더라도, "로그인 관련 변경"과 "오타 수정"을 별개의 커밋으로 깔끔하게 나눌 수 있습니다. 스테이징은 "이번 커밋에 무엇을 담을지" 선택하는 장바구니라고 생각하면 됩니다.

### 첫 커밋 만들기

```bash
# 현재 폴더를 Git 저장소로 초기화 (.git 폴더가 생성됨)
git init

# 작업 디렉터리의 모든 변경(.)을 스테이징 영역으로 올림
#  ·  .  은 현재 폴더 이하 전체를 의미
#  ·  특정 파일만 올리려면  git add 파일명  형태로 사용
git add .

# 스테이징된 내용을 하나의 커밋(스냅샷)으로 저장소에 기록
#  ·  -m  옵션은 커밋 메시지를 한 줄로 바로 입력
#  ·  메시지는 "무엇을 왜 바꿨는지" 명확하게 작성
git commit -m "첫 커밋"
```

### 상태 확인 습관 들이기

명령을 실행하기 전후로 현재 상태를 확인하는 습관이 중요합니다.

```bash
# 어떤 파일이 변경/스테이징되었는지 한눈에 확인 (가장 자주 쓰는 명령)
git status

# 지금까지의 커밋 이력을 보기
git log

# 한 줄로 간결하게 이력 보기 (커밋 해시 + 메시지)
git log --oneline
```

### 처음 한 번만 하는 설정

커밋에는 작성자 정보가 기록되므로, Git을 처음 쓸 때 이름과 이메일을 한 번 설정합니다.

```bash
# 모든 저장소에 적용되는 전역(--global) 사용자 이름 설정
git config --global user.name "홍길동"

# 전역 사용자 이메일 설정 (GitHub 가입 이메일과 맞추면 좋음)
git config --global user.email "you@example.com"
```

## 1.4 브랜치와 병합

브랜치(Branch)는 Git의 가장 강력한 기능입니다. **메인 코드는 건드리지 않고, 새로운 작업을 위한 평행 세계를 만드는 것**입니다. 새 기능을 실험하거나 버그를 고칠 때, 브랜치를 따로 만들어 작업하면 본래 코드가 망가질 걱정이 없습니다.

### 왜 브랜치가 필요한가

여러 명이 한 프로젝트를 작업하거나, 혼자라도 여러 기능을 동시에 진행할 때, 모두가 같은 줄기(main)에서 작업하면 서로의 작업이 뒤엉킵니다. 브랜치는 각 작업을 독립된 공간에 격리해 둡니다. 작업이 끝나면 그 결과를 다시 메인으로 **병합(merge)** 합니다. 실제 현업에서는 거의 모든 작업이 "브랜치 → 작업 → 병합" 흐름으로 이뤄집니다.

### 브랜치 만들고 합치기

```bash
# feature 라는 이름의 새 브랜치를 만들고 그 브랜치로 전환(-c = create)
#  ·  switch 는 비교적 최신 명령으로 브랜치 전환 전용
#  ·  예전 방식:  git checkout -b feature  (지금도 동작함)
git switch -c feature

# feature 브랜치에서 작업/커밋을 마친 뒤, 다시 main 브랜치로 돌아옴
git switch main

# feature 브랜치의 변경 내용을 현재 브랜치(main)로 병합
git merge feature
```

### 병합 결과를 그림으로 보기

```bash
# 커밋 이력을 그래프 형태로 시각화해서 보기
#  ·  --graph  브랜치 갈라짐/합쳐짐을 선으로 표현
#  ·  실무에서는  git log --oneline --graph --all  조합을 많이 사용
git log --graph
```

위 명령은 아래와 같은 형태로 브랜치가 갈라졌다가 합쳐진 흐름을 보여 줍니다.

```text
*  7f3a1c (main) 병합        ← feature를 main으로 병합한 커밋
|\
| *  f4e5d6 폼 추가          ← feature 브랜치에서 한 작업
|/
*  9f8e7d 초기 커밋          ← 갈라지기 전 공통 조상
```

### 자주 쓰는 브랜치 명령

```bash
# 현재 존재하는 브랜치 목록 보기 (*가 현재 브랜치)
git branch

# 병합이 끝나 더 이상 필요 없는 브랜치 삭제 (-d = delete)
git branch -d feature
```

### 초보자가 흔히 하는 실수

- main 브랜치에서 직접 작업하다가 실험이 꼬여 되돌리기 어려워지는 경우 — 새 작업은 항상 별도 브랜치에서 시작하세요.
- 브랜치를 만들기만 하고 `switch`로 전환하지 않아 여전히 main에서 작업하는 경우 — `git status`로 현재 브랜치를 항상 확인하세요.

## 1.5 병합 충돌 해결

브랜치를 병합할 때 **두 브랜치가 같은 파일의 같은 부분을 서로 다르게 고쳤다면** Git은 어느 쪽이 옳은지 스스로 판단할 수 없습니다. 이때 발생하는 것이 병합 충돌(merge conflict)입니다.

### 충돌은 오류가 아니다

가장 중요한 마음가짐입니다. **충돌은 에러가 아니라 정상적인 협업 과정**입니다. Git이 "여기 두 사람의 의견이 갈리니 사람이 결정해 주세요"라고 알려 주는 것뿐입니다. 당황하지 말고, Git이 표시한 부분을 정리한 뒤 다시 커밋하면 됩니다.

### 충돌 표시(conflict markers) 읽는 법

충돌이 나면 Git은 해당 파일 안에 아래와 같은 특수 표시를 넣어 줍니다.

```text
<<<<<<< HEAD
const title = "내 변경";
=======
const title = "상대 변경";
>>>>>>> feature
```

각 표시의 의미는 다음과 같습니다.

| 표시 | 의미 |
| --- | --- |
| `<<<<<<< HEAD` | 여기부터는 **현재 브랜치(내)** 의 코드 |
| `=======` | 두 코드를 나누는 경계선 |
| `>>>>>>> feature` | 여기까지는 **병합하려는 브랜치(상대)** 의 코드 |

### 해결 절차

1. 파일을 열어 `<<<<<<<`, `=======`, `>>>>>>>` 표시를 찾습니다.
2. 두 코드 중 원하는 것을 남기거나, 둘을 적절히 합칩니다.
3. **표시 줄(`<<<<<<<` 등)을 반드시 모두 삭제**합니다. 이 줄이 남아 있으면 코드가 깨집니다.
4. 정리한 파일을 다시 스테이징하고 커밋합니다.

```bash
# 충돌을 정리한 파일을 다시 스테이징하고 바로 커밋
#  ·  &&  앞 명령이 성공하면 뒤 명령 실행
#  ·  병합 중 커밋은 -m 없이 git commit 하면 자동 메시지가 채워짐
git add . && git commit

# 만약 병합을 아예 취소하고 충돌 이전 상태로 되돌리고 싶다면
git merge --abort
```

### 충돌을 줄이는 팁

- 작업 전에 `git pull`로 최신 코드를 받아 두면 충돌 가능성이 줄어듭니다.
- 한 브랜치를 너무 오래 방치하지 말고, 작은 단위로 자주 병합하세요.

## 1.6 GitHub & 원격 저장소

이제 내 컴퓨터(로컬)의 Git 저장소를 인터넷의 GitHub(원격, remote)와 연결할 차례입니다. 그러면 코드를 클라우드에 안전하게 보관하고, 다른 컴퓨터나 동료와 공유할 수 있습니다.

### GitHub 시작 3단계

| 단계 | 할 일 | 설명 |
| --- | --- | --- |
| 01 계정 만들기 | github.com에서 무료 가입 | 이메일만 있으면 무료로 시작 |
| 02 인터페이스 익히기 | Repositories · Issues · Pull requests · Actions | 각 탭의 역할을 눈에 익히기 |
| 03 첫 저장소 만들기 | New repository → README · .gitignore 추가 | README는 소개글, .gitignore는 제외할 파일 목록 |

- **README** 는 프로젝트를 소개하는 첫 화면 문서입니다.
- **.gitignore** 는 Git이 추적하지 않을 파일(예: 빌드 결과물, 비밀번호 파일, `node_modules`)을 적어 두는 파일입니다.

### 원격 저장소 연결과 동기화

연결의 기본 별칭(이름)은 관례적으로 **origin** 입니다. 코드를 올릴 땐 **push**, 받을 땐 **pull** 입니다.

```bash
# 로컬 저장소에 원격 저장소를 origin 이라는 별칭으로 연결
#  ·  <url> 자리에 GitHub 저장소 주소를 넣음 (https://... 또는 git@...)
git remote add origin <url>

# 로컬 main 브랜치 내용을 origin(원격)으로 올리기(push)
#  ·  -u  옵션은 main과 origin/main을 연결해 둠
#  ·  한 번 -u로 연결하면 이후엔 그냥 git push 만 해도 됨
git push -u origin main

# 원격의 최신 변경 사항을 받아와 현재 브랜치에 반영(pull = fetch + merge)
git pull

# 원격 저장소 전체를 내 컴퓨터로 통째로 복제(clone)
#  ·  새 프로젝트를 처음 받아올 때 사용
git clone <url>
```

### push / pull / clone 한눈에

| 명령 | 방향 | 언제 쓰나 |
| --- | --- | --- |
| `git push` | 로컬 → 원격 | 내 작업을 GitHub에 올릴 때 |
| `git pull` | 원격 → 로컬 | 동료의 작업을 받아올 때 |
| `git clone` | 원격 → 로컬(최초) | 프로젝트를 처음 통째로 받을 때 |
| `git remote add` | 연결 설정 | 로컬과 원격을 처음 잇는 1회성 작업 |

### 초보자가 흔히 겪는 문제

- `git push`가 거부될 때 — 원격에 내가 모르는 변경이 있는 경우입니다. 먼저 `git pull`로 받아온 뒤 다시 push 하세요.
- 비밀번호 대신 토큰 요구 — GitHub는 비밀번호 대신 개인 액세스 토큰(PAT)이나 SSH 키를 사용합니다.

## 1.7 GitHub 협업: Fork · Pull Request · Issue

여러 사람이 함께 일하거나 오픈소스에 기여할 때 따르는 **표준 흐름**이 있습니다.

```text
Fork  →  Branch  →  Push  →  Pull Request  →  Review & Merge
```

### 핵심 용어

| 용어 | 의미 | 비유 |
| --- | --- | --- |
| Fork | 원본 저장소를 **내 계정으로 복사** | 책을 통째로 내 책상에 복사해 오기 |
| Branch | 작업할 독립 가지 만들기 | 작업 공간 분리 |
| Pull Request (PR) | "내 변경을 합쳐 주세요"라는 **제안 + 코드 리뷰** 요청 | 검토 요청서 제출 |
| Issue | 버그·기능 요청·할 일을 기록하는 게시판 | 할 일 메모판 |

### 표준 기여 흐름 자세히

1. **Fork** — 직접 수정 권한이 없는 남의 저장소를 내 계정으로 복사합니다.
2. **Clone & Branch** — 내 Fork를 로컬에 받아오고, 작업용 브랜치를 만듭니다.
3. **작업 & Push** — 코드를 고치고 커밋한 뒤, 내 Fork로 push 합니다.
4. **Pull Request** — 원본 저장소에 "내 변경을 합쳐 달라"고 PR을 보냅니다.
5. **Review & Merge** — 관리자가 코드를 리뷰하고, 괜찮으면 병합합니다.

```bash
# Fork한 내 저장소를 로컬로 복제
git clone <내-fork-url>

# 작업용 브랜치 생성 및 전환
git switch -c fix-login-bug

# (코드 수정 후) 변경 스테이징 및 커밋
git add .
git commit -m "로그인 유효성 검사 추가"

# 내 Fork(origin)로 브랜치 push → GitHub에서 PR 버튼이 나타남
git push -u origin fix-login-bug
```

### Issue 활용

Issue는 단순 버그 신고를 넘어, 프로젝트의 **할 일 목록이자 토론 공간**입니다. "이런 기능이 있으면 좋겠다", "여기 버그가 있다" 등을 기록하고 번호로 추적합니다. 커밋 메시지나 PR에서 `#12`처럼 이슈 번호를 적으면 자동으로 연결됩니다.

### 권한(Permission) 단계

협업자에게는 저장소 권한을 단계적으로 부여합니다.

| 권한 | 할 수 있는 일 |
| --- | --- |
| Read | 코드 보기, 이슈 작성 |
| Write | 직접 push, 브랜치 생성 |
| Admin | 설정 변경, 협업자 관리, 저장소 삭제 |

권한은 **Read → Write → Admin** 순으로 강력해집니다. 최소 권한 원칙에 따라, 필요한 만큼만 부여하는 것이 안전합니다.

## 1.8 좋은 커밋 메시지와 오픈소스 라이선스

### 좋은 커밋 메시지 쓰기

커밋 메시지는 미래의 나와 동료에게 보내는 메모입니다. 시간이 지나면 "이걸 왜 바꿨더라?" 하게 되는데, 좋은 메시지가 그 답을 줍니다.

| 나쁜 예 ✕ | 좋은 예 ✓ |
| --- | --- |
| "수정" | "로그인 유효성 검사 추가" |
| "asdf" | "결제 반올림 오류 수정" |
| "버그 고침 (또)" | "장바구니 수량 0 입력 방지" |

좋은 커밋 메시지의 원칙은 다음과 같습니다.

- **제목은 50자 이내**로 간결하게.
- **명령형**으로 작성: "추가했음"이 아니라 "추가" 처럼.
- **무엇을, 왜** 바꿨는지가 드러나게.
- **커밋 하나엔 논리적 변경 하나만** 담기. 서로 무관한 변경을 한 커밋에 섞지 않기.

```bash
# 좋은 커밋 메시지 예시 (제목 = 무엇을 왜)
git commit -m "로그인 유효성 검사 추가"

# 본문까지 자세히 쓰려면 -m 을 두 번 사용 (첫 번째=제목, 두 번째=본문)
git commit -m "결제 반올림 오류 수정" -m "소수점 셋째 자리에서 발생하던 1원 오차를 반올림 방식 변경으로 해결"
```

### 대표 오픈소스 라이선스

코드를 공개할 때는 다른 사람이 어떻게 사용할 수 있는지를 정하는 **라이선스**가 필요합니다. 라이선스가 없으면 법적으로는 "아무도 함부로 쓸 수 없는" 상태가 됩니다.

| 라이선스 | 자유도 | 핵심 내용 | 적합한 경우 |
| --- | --- | --- | --- |
| MIT | 매우 높음 | 저작권 표시만 유지하면 상업적 사용·수정·재배포 자유 | 가장 널리 쓰임, 부담 없이 공유 |
| Apache 2.0 | 높음 | MIT와 비슷하되 **특허권 관련 보호 조항** 추가 | 기업·특허 이슈가 중요한 프로젝트 |
| GPL | 제한적(카피레프트) | 이 코드를 쓴 **결과물도 같은 라이선스로 공개**해야 함 | 파생물도 오픈소스로 유지하고 싶을 때 |

핵심 구분은 **"내 코드를 쓴 결과물도 공개해야 하는가"** 입니다. MIT와 Apache는 그럴 의무가 없지만(자유로움), GPL은 파생 결과물도 같은 조건으로 공개하도록 강제합니다. 이를 **카피레프트(copyleft)** 라고 부릅니다.

## 1.9 고급 기능과 트러블슈팅

기본기를 익혔다면, 실무에서 자주 마주치는 아래 기능들을 알아 두면 훨씬 유연하게 대처할 수 있습니다.

### 알아두면 좋은 4가지

| 기능 | 한 줄 설명 | 주의 |
| --- | --- | --- |
| `git stash` | 작업을 잠시 치워 두고 나중에 되돌리기 | 커밋하기 애매한 중간 작업 보관용 |
| `git rebase` | 이력을 직선으로 깔끔하게 정리 | **공유 브랜치에서는 금지** |
| git hooks | 특정 이벤트 발생 시 자동 실행되는 스크립트 | 커밋 전 자동 검사 등에 활용 |
| blame / bisect | 누가 바꿨나 추적 / 버그 만든 커밋 찾기 | 디버깅에 강력 |

### git stash — 작업을 잠시 치워 두기

지금 하던 작업을 커밋하기는 애매한데, 갑자기 다른 브랜치로 옮겨 급한 일을 처리해야 할 때 사용합니다.

```bash
# 현재 작업 중인 변경을 임시 저장소(stash)에 치워 두고 작업 디렉터리를 깨끗하게
git stash

# 치워 둔 작업을 다시 꺼내 와 복원 (꺼내면서 목록에서 제거)
git stash pop

# 치워 둔 stash 목록 보기
git stash list
```

### git rebase — 이력을 직선으로 정리

병합 대신 커밋 이력을 한 줄로 깔끔하게 만들고 싶을 때 사용합니다.

```bash
# 현재 브랜치의 커밋들을 main 위로 재배치해 이력을 직선으로 정리
#  ·  여러 명이 공유하는 브랜치에서는 절대 하지 말 것(이력이 바뀌어 혼란)
git rebase main
```

### 위험 상황 되돌리기

이미 공개(push)한 커밋을 안전하게 되돌리려면 이력을 지우는 reset 대신 **revert**를 씁니다.

```bash
# 특정 커밋의 변경을 "취소하는 새 커밋"을 만들어 안전하게 되돌림
#  ·  이력을 지우지 않으므로 공유 브랜치에서도 안전
git revert <커밋해시>
```

### 당황하지 마세요 — reflog로 잃은 커밋 되살리기

실수로 reset하거나 브랜치를 지워 커밋이 사라진 것처럼 보여도, **reflog가 거의 모든 이동 기록을 갖고 있습니다.** Git에서 진짜로 무언가를 완전히 잃기는 생각보다 어렵습니다.

```bash
# HEAD가 움직인 모든 기록(브랜치 전환, 커밋, reset 등)을 시간순으로 보기
git reflog

# reflog에서 찾은 잃어버린 커밋 해시로 복귀
#  ·  a1b2c3d 자리에 reflog에서 확인한 해시를 넣음
git checkout a1b2c3d
```

### blame / bisect — 추적과 디버깅

```bash
# 파일의 각 줄을 누가, 언제, 어떤 커밋에서 바꿨는지 표시
git blame 파일명

# 이진 탐색으로 버그를 처음 만든 커밋을 빠르게 찾기 시작
git bisect start
```

## 1.10 명령어 치트시트와 다음 단계

자주 쓰는 명령을 용도별로 정리했습니다. 외우려 하기보다, 필요할 때 찾아 쓰며 손에 익히세요.

### 시작 & 커밋 & 이력

| 시작 | 커밋 | 이력 |
| --- | --- | --- |
| `git init` | `git add .` | `git status` |
| `git clone <url>` | `git commit -m "…"` | `git log --oneline` |

### 브랜치 & 원격 & 되돌리기

| 브랜치 | 원격 | 되돌리기 |
| --- | --- | --- |
| `git switch -c <name>` | `git pull` | `git revert <c>` |
| `git merge <name>` | `git push` | `git stash pop` |

### 전체 치트시트 (한 번에 보기)

```bash
# ── 시작 ──────────────────────────────
git init                 # 현재 폴더를 Git 저장소로 초기화
git clone <url>          # 원격 저장소를 통째로 복제

# ── 변경 & 커밋 ──────────────────────
git status               # 현재 변경/스테이징 상태 확인
git add .                # 모든 변경을 스테이징
git commit -m "…"        # 스테이징된 변경을 커밋으로 기록

# ── 이력 보기 ────────────────────────
git log --oneline        # 커밋 이력을 한 줄씩 간결하게
git log --graph          # 브랜치 흐름을 그래프로

# ── 브랜치 & 병합 ────────────────────
git switch -c <name>     # 새 브랜치 생성 + 전환
git switch <name>        # 기존 브랜치로 전환
git merge <name>         # 지정 브랜치를 현재 브랜치로 병합
git branch -d <name>     # 병합 끝난 브랜치 삭제

# ── 원격(GitHub) ─────────────────────
git remote add origin <url>   # 원격 저장소 연결
git push -u origin main       # 로컬 → 원격 올리기(최초 연결)
git push                      # 이후 변경 올리기
git pull                      # 원격 변경 받아오기

# ── 되돌리기 & 구조용 ─────────────────
git revert <커밋해시>     # 커밋을 취소하는 새 커밋(안전)
git stash                # 작업 임시 보관
git stash pop            # 보관한 작업 복원
git reflog               # 모든 이동 기록(잃은 커밋 찾기)
```

2. **오픈소스의 `good first issue` 라벨이 붙은 이슈를 하나 골라 해결**해 보세요. 초보자를 위해 일부러 쉽게 만들어 둔 진입점입니다.
3. **막히면 공식 문서와 커뮤니티를 적극 활용**하세요. Git 공식 문서, GitHub Docs, Stack Overflow가 든든한 동료가 되어 줍니다.

작게 시작하고, 꾸준히 커밋하세요. Happy committing!
