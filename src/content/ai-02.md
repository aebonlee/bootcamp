## 2.1 파이프라인(pipeline)으로 모델 불러오기

Hugging Face transformers 라이브러리에서 `pipeline()`은 가장 핵심이 되는 고수준(high-level) API입니다. 수십 줄의 초기화 코드를 단 한 줄로 압축해 주며, 모델 로딩, 토크나이징, 추론(inference), 결과 디코딩까지 모든 과정을 내부적으로 처리합니다.

### (1) Hugging Face pipeline()의 개념과 작동 원리

`pipeline()`은 소프트웨어 공학의 '파사드 패턴(Facade Pattern)'을 AI 추론 흐름에 적용한 것입니다. 사용자는 복잡한 내부 구조를 전혀 몰라도 표준화된 인터페이스만으로 강력한 AI 기능을 바로 사용할 수 있습니다.

`pipeline()`이 내부적으로 수행하는 일련의 작업 흐름은 다음과 같습니다.

| 단계 | 내용 | 담당 클래스 |
| --- | --- | --- |
| 1. 모델 식별 | 태스크명 또는 모델명으로 Hub에서 적합한 모델을 선택 | `AutoConfig` |
| 2. 토크나이저 로드 | 텍스트를 숫자 ID 시퀀스로 변환하는 토크나이저 초기화 | `AutoTokenizer` |
| 3. 모델 로드 | 사전 학습된 가중치(weights) 및 구조를 로컬 캐시에 저장 후 로드 | `AutoModel` |
| 4. 전처리 | 입력 텍스트를 토큰 ID, attention mask, token_type_ids로 변환 | `Tokenizer.__call__` |
| 5. 추론 (Inference) | GPU/CPU에서 forward pass를 실행하고 로짓(logits)을 계산 | `Model.forward()` |
| 6. 후처리 | 숫자 출력을 사람이 읽을 수 있는 레이블/텍스트로 변환 | `PostProcessor` |

이 전체 흐름을 아래의 단 두 줄 코드로 실행할 수 있습니다.

```python
# transformers 라이브러리에서 고수준 API인 pipeline 함수를 가져옵니다
from transformers import pipeline

# 'sentiment-analysis'(감성 분석) 태스크명만 넘기면
# Hugging Face가 이 태스크에 맞는 기본 모델/토크나이저를 자동 다운로드·로드합니다.
# 반환값 classifier는 "텍스트를 넣으면 분석 결과를 주는" 호출 가능한 객체입니다.
classifier = pipeline('sentiment-analysis')
# 분석할 문장을 인자로 넘기면 내부에서 토크나이징→추론→후처리가 한 번에 실행됩니다
result = classifier('I love using Hugging Face!')
# 결과는 딕셔너리 리스트 형태로 반환되므로 그대로 출력해 확인합니다
print(result)
# 출력: [{'label': 'POSITIVE', 'score': 0.9998}]  ← label: 예측 감성, score: 확신도(0~1)
```

*[코드 2-1] pipeline() 가장 기본적인 사용 예*

`pipeline()`의 첫 번째 인자에는 태스크명(task name)을 문자열로 전달합니다. 태스크명에 따라 Hugging Face가 해당 태스크에 최적화된 기본 모델을 자동으로 선택하여 로드합니다. 물론 특정 모델을 직접 지정할 수도 있습니다.

```python
# 특정 모델을 명시적으로 지정하는 방법
# task 인자로 수행할 작업을, model 인자로 사용할 정확한 모델 이름을 지정합니다.
# 기본 추천 모델 대신 원하는 모델을 고정하고 싶을 때 이렇게 작성합니다.
classifier = pipeline(
    task='sentiment-analysis',  # 수행할 NLP 태스크 종류
    model='distilbert-base-uncased-finetuned-sst-2-english'  # SST-2로 파인튜닝된 영어 감성 분석 모델
)
```

*[코드 2-2] 모델 이름을 명시적으로 지정하는 방법*

> 처음 `pipeline()`을 실행하면 모델 파일을 인터넷에서 다운로드합니다. 모델 크기에 따라 수백 MB ~ 수 GB의 용량이 필요할 수 있으므로, 안정적인 인터넷 환경에서 실행하세요. 다운로드된 파일은 로컬 캐시(`~/.cache/huggingface/`)에 저장되어 이후 실행 시 재활용됩니다.

### (2) 텍스트 분류, 텍스트 생성, 번역 등 다양한 태스크 지원

`pipeline()`이 지원하는 태스크의 범위는 매우 넓습니다. 자연어 처리(NLP), 컴퓨터 비전(CV), 음성 처리(Audio)에 이르기까지 현재 70개 이상의 태스크를 지원하며, 이 수는 계속 확장되고 있습니다.

아래 표는 자연어 처리 분야에서 가장 자주 활용되는 pipeline 태스크를 정리한 것입니다.

| 태스크 문자열 | 설명 | 대표 활용 예 |
| --- | --- | --- |
| `sentiment-analysis` | 텍스트의 긍/부정 감성 분류 | 리뷰 분석, 고객 피드백 처리 |
| `text-generation` | 프롬프트 뒤에 이어지는 텍스트 생성 | 챗봇, 글쓰기 보조 도구 |
| `text2text-generation` | 입력 텍스트를 다른 텍스트로 변환 | 요약, 번역, 질의응답 |
| `summarization` | 긴 문서를 짧게 요약 | 뉴스 요약, 보고서 요약 |
| `translation` | 언어 간 번역 | 영한 번역, 다국어 서비스 |
| `question-answering` | 문맥에서 질문에 대한 답 추출 | FAQ 자동화, 검색 엔진 |
| `fill-mask` | 마스킹된 토큰 예측 | 언어 모델 평가, 데이터 증강 |
| `ner` (token-classification) | 개체명 인식 (인물, 장소, 조직 등) | 정보 추출, 문서 파싱 |
| `zero-shot-classification` | 학습 없이 임의 레이블로 분류 | 빠른 프로토타이핑 |
| `text-classification` | 다중 클래스 텍스트 분류 | 카테고리 분류, 스팸 필터 |

각 태스크는 전용 파이프라인 클래스로 구현되어 있으며, `pipeline()` 함수가 태스크명에 따라 적절한 클래스 인스턴스를 반환합니다. 예를 들어 `sentiment-analysis`는 내부적으로 `TextClassificationPipeline` 클래스를 반환합니다.

```python
# pipeline 함수 하나로 여러 종류의 태스크를 생성할 수 있습니다
from transformers import pipeline

# 태스크별 pipeline 초기화 예시
# 첫 번째 인자(태스크 문자열)만 바꾸면 완전히 다른 AI 기능을 얻습니다.

# 1) 감성 분석: 텍스트의 긍정/부정 분류 (모델 미지정 → 기본 모델 자동 선택)
sentiment_pipe = pipeline('sentiment-analysis')

# 2) 텍스트 요약: 긴 글을 짧게 줄임 (BART를 CNN 뉴스로 파인튜닝한 요약 전용 모델 지정)
summarizer = pipeline('summarization', model='facebook/bart-large-cnn')

# 3) 번역 (영어 → 프랑스어): 태스크명에 언어쌍(en_to_fr)을 포함, Helsinki-NLP의 영-불 번역 모델 사용
translator = pipeline('translation_en_to_fr', model='Helsinki-NLP/opus-mt-en-fr')

# 4) 개체명 인식: 인물·장소·조직 등 추출. grouped_entities=True는 쪼개진 토큰을 하나의 개체로 묶어 줍니다
ner = pipeline('ner', grouped_entities=True)

# 5) Zero-shot 분류: 학습하지 않은 임의의 레이블로 분류. MNLI로 학습한 BART가 함의(entailment) 추론에 사용됩니다
zero_shot = pipeline('zero-shot-classification', model='facebook/bart-large-mnli')
```

이제 기본 감정 분석 파이프라인을 실행합니다. 처음 실행 시 모델이 다운로드되므로 시간이 다소 걸립니다.

```python
from transformers import pipeline

# 감정 분석 파이프라인 초기화
# 기본 모델: distilbert-base-uncased-finetuned-sst-2-english
# 처음 호출 시 모델 파일을 인터넷에서 받아 로컬 캐시에 저장합니다(시간 소요).
classifier = pipeline('sentiment-analysis')

# 단일 문장 분석: 문자열 하나를 넘기면 결과 1개가 담긴 리스트가 반환됩니다
result = classifier('This movie is absolutely wonderful!')
print(result)
# 출력: [{'label': 'POSITIVE', 'score': 0.9998}]

# 여러 문장 동시 분석 (배치 처리)
# 리스트를 한 번에 넘기면 모델이 묶어서(batch) 처리해 한 문장씩 부르는 것보다 빠릅니다.
texts = [
    'I really enjoyed the lecture today.',
    'The weather is terrible and I feel miserable.',
    'The product quality is acceptable, nothing special.',
]
# 입력 순서와 동일한 순서로 결과 리스트가 반환됩니다
results = classifier(texts)
# zip으로 원본 문장과 결과를 짝지어 보기 좋게 출력합니다
# {res["label"]:8s}: 레이블을 8칸 폭으로 정렬, {res["score"]:.4f}: 점수를 소수점 4자리로 표시
for text, res in zip(texts, results):
    print(f'[{res["label"]:8s} {res["score"]:.4f}] {text}')
```

> `score` 값은 모델이 해당 레이블에 부여한 확신도(confidence)입니다. 1.0에 가까울수록 모델이 해당 분류에 확신하고 있음을 의미합니다. 세 번째 문장처럼 중립적인 표현은 0.6 수준으로 비교적 낮은 점수를 보입니다.

`pipeline()`의 반환값 구조를 좀 더 자세히 살펴봅니다. 반환값은 항상 Python 딕셔너리 리스트(`List[Dict]`) 형태입니다.

| 키(Key) | 타입 | 설명 |
| --- | --- | --- |
| `label` | `str` | 예측된 클래스 레이블 (예: 'POSITIVE', 'NEGATIVE') |
| `score` | `float` | 해당 레이블에 대한 확신도 (0.0 ~ 1.0 사이) |

`return_all_scores=True` 옵션을 사용하면 모든 레이블에 대한 점수를 한 번에 확인할 수 있습니다.

```python
# 모든 레이블의 점수 확인
# 기본값은 최고 점수 1개만 반환하지만, 아래 옵션을 켜면 모든 레이블의 점수를 받습니다.
classifier = pipeline(
    'sentiment-analysis',
    return_all_scores=True        # 모든 레이블 점수 반환 / 또는 top_k=None (신버전)
)

# 중립적인 문장이라 POSITIVE/NEGATIVE 점수가 비교적 비슷하게 나옵니다
result = classifier('The experience was just okay.')
print(result)
# 출력:
# [[{'label': 'NEGATIVE', 'score': 0.3241},
#   {'label': 'POSITIVE', 'score': 0.6759}]]
```

### (4) GPU/CPU 선택 방법 및 속도 비교

딥러닝 모델의 추론 속도는 실행 장치에 따라 크게 달라집니다. Hugging Face `pipeline()`은 `device` 파라미터를 통해 실행 장치를 간편하게 지정할 수 있습니다.

| device 값 | 의미 | 적용 대상 |
| --- | --- | --- |
| `-1` (기본값) | CPU에서 실행 | GPU 없는 환경, 소규모 모델 |
| `0` | 첫 번째 GPU (cuda:0) 에서 실행 | 단일 GPU 서버/데스크톱 |
| `1` | 두 번째 GPU (cuda:1) 에서 실행 | 멀티 GPU 환경 |
| `'mps'` | Apple Silicon (M1/M2/M3) GPU 사용 | Mac 환경 |

```python
import torch
from transformers import pipeline
import time  # 실행 시간 측정용 표준 라이브러리

# 장치 자동 선택 함수: 환경에 따라 적절한 device 값을 반환합니다
def get_device():
    # 1순위: NVIDIA GPU(CUDA)가 있으면 GPU 이름을 출력하고 0(첫 GPU) 반환
    if torch.cuda.is_available():
        print(f'GPU: {torch.cuda.get_device_name(0)}')
        return 0    # 첫 번째 CUDA GPU
    # 2순위: Apple Silicon(M1/M2/M3)의 MPS 가속을 사용할 수 있으면 'mps' 반환
    elif hasattr(torch.backends, 'mps') and torch.backends.mps.is_available():
        print('Apple Silicon MPS 사용')
        return 'mps'
    # 그 외에는 CPU(-1) 사용
    else:
        print('CPU 사용')
        return -1

# 현재 환경에 맞는 장치 값을 미리 구해 둡니다
device = get_device()

# CPU로 파이프라인 초기화 (device=-1은 CPU 사용을 의미)
cpu_pipe = pipeline('sentiment-analysis', device=-1)

# GPU/MPS가 있다면(-1이 아니면) GPU용 파이프라인도 따로 초기화
if device != -1:
    gpu_pipe = pipeline('sentiment-analysis', device=device)

# 배치 크기 100으로 속도 비교 테스트
# 동일한 문장 100개를 만들어 처리량(문장/초)을 측정합니다
test_texts = ['This is a test sentence.'] * 100

# CPU 속도 측정
start = time.time()              # 시작 시각 기록
cpu_results = cpu_pipe(test_texts)  # 100개 문장을 한 번에 추론
cpu_time = time.time() - start   # 경과 시간(초) 계산
# 전체 문장 수를 경과 시간으로 나눠 초당 처리 문장 수를 출력합니다
print(f'CPU 처리 시간: {cpu_time:.2f}초 ({len(test_texts)/cpu_time:.1f} 문장/초)')
```

### (5) 파이프라인 내부 동작 구조 간단 이해 (AutoTokenizer, AutoModel)

`pipeline()`이 내부적으로 어떤 일을 수행하는지 이해하면, 나중에 커스텀 파이프라인을 만들거나 세밀한 제어가 필요할 때 큰 도움이 됩니다. `pipeline()`의 핵심 구성 요소는 `AutoTokenizer`와 `AutoModel`(혹은 태스크별 AutoModel) 두 가지입니다.

아래 코드는 `pipeline()`과 동일한 작업을 `AutoTokenizer`와 `AutoModel`을 직접 사용하여 수행하는 예입니다. 두 접근 방식의 결과는 동일합니다.

```python
import torch
# AutoTokenizer: 모델에 맞는 토크나이저 자동 선택, AutoModelFor...: 분류 헤드가 달린 모델 자동 선택
from transformers import AutoTokenizer, AutoModelForSequenceClassification

# 1. 모델명 지정 (Hub의 모델 식별자)
model_name = 'distilbert-base-uncased-finetuned-sst-2-english'

# 2. 토크나이저 로드: 해당 모델이 학습 때 쓴 어휘/규칙대로 텍스트를 토큰화하도록 준비
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 3. 모델 로드: 사전 학습된 가중치와 분류 헤드를 함께 불러옵니다
model = AutoModelForSequenceClassification.from_pretrained(model_name)
model.eval()    # 추론 모드로 전환 (dropout 등 학습 전용 동작 비활성화 → 결과 재현성 확보)

# 4. 입력 텍스트 전처리
text = 'I absolutely love this product!'
# tokenizer가 텍스트를 토큰 ID로 변환, return_tensors='pt'로 PyTorch 텐서 형태 반환
inputs = tokenizer(text, return_tensors='pt')  # PyTorch 텐서 반환
print('Input IDs:', inputs['input_ids'])        # 각 토큰의 숫자 ID 시퀀스
print('Attention Mask:', inputs['attention_mask'])  # 실제 토큰=1, 패딩=0을 나타내는 마스크

# 5. 추론 실행
with torch.no_grad():    # 그래디언트 계산 비활성화 (메모리 절약 + 추론 속도 향상)
    outputs = model(**inputs)   # **inputs로 input_ids, attention_mask를 모델에 전달
    logits = outputs.logits     # 정규화 전 원시 점수(클래스별 raw score)
    print('Logits:', logits)

# 6. 후처리: softmax → 확률 변환
# softmax는 logits를 합이 1인 확률 분포로 변환합니다 (dim=-1: 마지막 차원=클래스 축 기준)
probs = torch.nn.functional.softmax(logits, dim=-1)
predicted_class = logits.argmax(-1).item()   # 가장 점수 높은 클래스의 인덱스를 정수로 추출
labels = model.config.id2label               # 인덱스→사람이 읽는 레이블 매핑 딕셔너리
# 예측 레이블과 그 확률을 함께 출력
print(f'결과: {labels[predicted_class]} ({probs[0][predicted_class]:.4f})')
```

*[코드 2-9] AutoTokenizer + AutoModel로 pipeline 내부 구현 직접 재현*

> `AutoModelForSequenceClassification`은 텍스트 분류에 특화된 자동 모델 선택 클래스입니다. 태스크에 따라 `AutoModelForSeq2SeqLM`(요약/번역), `AutoModelForCausalLM`(텍스트 생성), `AutoModelForTokenClassification`(NER)처럼 다양한 변형이 존재합니다.

토크나이저의 출력을 직접 확인해 보면 텍스트가 어떻게 숫자 시퀀스로 변환되는지 이해할 수 있습니다.

```python
from transformers import AutoTokenizer

# BERT 기본 모델의 토크나이저 로드 (소문자화 처리하는 uncased 버전)
tokenizer = AutoTokenizer.from_pretrained('bert-base-uncased')
text = 'Hello, Hugging Face!'

# 토큰 확인: 문장을 모델이 다루는 토큰(단어 조각) 단위로 쪼갭니다
tokens = tokenizer.tokenize(text)
print('Tokens:', tokens)
# 출력: ['hello', ',', 'hugging', 'face', '!']

# 토큰 → ID 변환: 각 토큰을 어휘 사전(vocab)의 정수 ID로 매핑합니다
token_ids = tokenizer.convert_tokens_to_ids(tokens)
print('Token IDs:', token_ids)
# 출력: [7592, 1010, 17662, 2227, 999]

# 전체 인코딩 (특수 토큰 [CLS], [SEP] 포함)
# tokenizer를 직접 호출하면 위 두 단계에 더해 문장 시작/끝 특수 토큰까지 자동으로 붙여 줍니다
encoded = tokenizer(text)
print('Encoded IDs:', encoded['input_ids'])
# 출력: [101, 7592, 1010, 17662, 2227, 999, 102]
#        ^^^                              ^^^
#        [CLS]                          [SEP]
```

## 2.2 텍스트 생성 실습 (GPT-2, KoGPT)

### (1) text-generation 태스크란?

텍스트 생성(text-generation)은 주어진 프롬프트(prompt) 텍스트를 입력으로 받아 그 뒤에 이어지는 텍스트를 자동으로 생성하는 태스크입니다. GPT(Generative Pre-trained Transformer) 계열 모델이 이 태스크에 특화되어 있습니다.

텍스트 생성 모델은 언어 모델링(Language Modeling) 사전 학습 방식을 따릅니다. 즉, 엄청난 양의 텍스트에서 '다음 단어를 예측하는 능력'을 학습한 모델입니다. 이 능력이 고도화되면 문장, 단락, 심지어 코드나 시까지 생성할 수 있게 됩니다.

#### 텍스트 생성 모델의 학습 방식

- 언어 모델 사전 학습 (Causal Language Modeling):
  - 입력: '오늘 날씨가 매우 좋아서 공원에'
  - 목표: '놀러' 를 예측
  - 다음 입력: '오늘 날씨가 매우 좋아서 공원에 놀러'
  - 목표: '가고' 를 예측 ... (반복)
- 이렇게 수백억 개의 문장에서 학습하면 언어의 패턴, 문법, 세계 지식까지 학습

Hugging Face에서 text-generation 파이프라인을 사용하는 가장 기본적인 형태는 다음과 같습니다.

```python
from transformers import pipeline

# GPT-2 기반 텍스트 생성 파이프라인
# 'text-generation' 태스크 + 'gpt2' 모델 → 프롬프트 뒤를 자동으로 이어 쓰는 객체
generator = pipeline('text-generation', model='gpt2')

# 프롬프트 입력 → 텍스트 생성
prompt = 'Artificial intelligence will change the world by'
# max_new_tokens=80: 프롬프트 뒤에 새로 생성할 토큰 수를 최대 80개로 제한
outputs = generator(prompt, max_new_tokens=80)

# 반환값은 리스트, 각 항목의 'generated_text'에 (프롬프트+생성문) 전체가 들어 있습니다
print(outputs[0]['generated_text'])
```

### (2) Hugging Face Hub에서 생성형 모델 검색하기

Hugging Face Hub(https://huggingface.co/models)는 전 세계 연구자와 개발자들이 공유한 수십만 개의 모델을 검색하고 다운로드할 수 있는 플랫폼입니다. 텍스트 생성 모델을 검색하는 방법을 알아봅니다.

Hub에서 모델을 검색할 때는 Tasks 필터를 'Text Generation'으로 설정하고, 사용하려는 언어에 맞게 Language 필터를 추가 설정합니다. 한국어 생성 모델을 찾으려면 Language를 'Korean'으로 설정합니다.

#### Hub 검색 팁

- 라이브러리 필터: PyTorch 또는 TensorFlow로 좁힐 수 있음
- Sort by: Trending (인기순), Most Downloads, Recently Updated 등
- Model card: 각 모델 페이지에는 사용법, 라이선스, 성능 지표가 정리됨
- 'Use in Transformers' 버튼: 클릭하면 `from_pretrained()` 코드가 자동 생성됨

Python 코드로 Hub API를 통해 특정 태스크의 모델 목록을 조회할 수도 있습니다.

```python
# pip install huggingface_hub 설치 필요
# list_models: Hub의 모델 목록을 프로그램으로 검색하는 함수
from huggingface_hub import list_models

# 한국어 텍스트 생성 모델 목록 조회
# 필터 조건을 인자로 넘기면 조건에 맞는 모델 메타데이터를 받아옵니다.
models = list_models(
    task='text-generation',  # 태스크로 필터링
    language='ko',           # 한국어 모델만
    sort='downloads',        # 다운로드 수 기준 정렬
    direction=-1,            # 내림차순 (다운로드 많은 순)
    limit=10                 # 상위 10개만 가져오기
)

print('한국어 텍스트 생성 모델 TOP 10:')
# enumerate(..., 1): 1번부터 번호를 매겨 순회, m.modelId는 모델의 전체 이름
for i, m in enumerate(models, 1):
    print(f' {i:2d}. {m.modelId}')  # {i:2d}: 번호를 2칸 폭으로 정렬
```

### (3) 영어 모델(GPT-2)과 한국어 모델(KoGPT) 비교 실습

GPT-2는 OpenAI가 공개한 영어 언어 모델로, 파라미터 수에 따라 Small(124M), Medium(355M), Large(774M), XL(1.5B) 네 가지 크기가 있습니다. 한국어 처리를 위해서는 한국어 데이터로 학습된 KoGPT 계열 모델을 사용해야 합니다.

| 모델명 | 파라미터 수 | 학습 언어 | 특징 |
| --- | --- | --- | --- |
| `gpt2` | 124M | 영어 | OpenAI 공개 기본 모델, 빠른 실행 |
| `gpt2-medium` | 355M | 영어 | GPT-2 중간 크기, 균형잡힌 성능 |
| `gpt2-large` | 774M | 영어 | 고품질 생성, 느린 실행 |
| `skt/kogpt2-base-v2` | 125M | 한국어 | SKT 공개 한국어 GPT-2 |
| `kakaobrain/kogpt` | 6B | 한국어 | 카카오브레인 대형 한국어 GPT |
| `beomi/kcbert-base` | 110M | 한국어 | 분류 태스크에 강한 한국어 BERT |

먼저 GPT-2(영어)로 텍스트를 생성해 봅니다.

```python
from transformers import pipeline

# ── GPT-2 (영어) 텍스트 생성 ──────────────────────────────
gpt2_gen = pipeline('text-generation', model='gpt2')

prompt_en = 'The future of artificial intelligence is'
result_en = gpt2_gen(
    prompt_en,
    max_new_tokens=100,          # 새로 생성할 최대 토큰 수
    num_return_sequences=2,      # 서로 다른 생성 결과를 2개 반환
    temperature=0.8,             # 창의성 조절 (낮을수록 일관성, 높을수록 다양성)
    do_sample=True,              # 확률 샘플링 사용 (True여야 temperature 등이 효과를 냄)
    pad_token_id=50256           # GPT-2는 패딩 토큰이 없어 eos 토큰 ID(50256)를 패딩으로 지정 → 경고 방지
)

print('=== GPT-2 생성 결과 ===')
# num_return_sequences=2이므로 결과가 2개, 각각 번호를 붙여 출력
for i, res in enumerate(result_en, 1):
    print(f'\n--- 생성 #{i} ---')
    print(res['generated_text'])
```

*[코드 2-13] GPT-2 영어 텍스트 생성*

이제 KoGPT2로 한국어 텍스트를 생성합니다. KoGPT2는 SKT에서 공개한 한국어 GPT-2 모델로, 위키피디아 및 다양한 한국어 웹 문서로 학습되었습니다.

```python
# AutoModelForCausalLM: 다음 토큰을 예측하는 자기회귀(생성형) 모델용 클래스
from transformers import pipeline, AutoTokenizer, AutoModelForCausalLM

# ── KoGPT2 (한국어) 텍스트 생성 ───────────────────────────
# KoGPT2는 특수한 토크나이저를 사용하므로 직접 로드 권장
model_name = 'skt/kogpt2-base-v2'

# 토크나이저를 로드하면서 KoGPT2가 요구하는 특수 토큰을 명시적으로 지정합니다
tokenizer = AutoTokenizer.from_pretrained(
    model_name,
    bos_token='</s>',    # 문장 시작(begin-of-sentence) 토큰
    eos_token='</s>',    # 문장 끝(end-of-sentence) 토큰
    unk_token='<unk>',   # 사전에 없는 단어(unknown) 토큰
    pad_token='<pad>',   # 길이 맞춤용 패딩 토큰
    mask_token='<mask>', # 마스킹 토큰
)

# 생성형 모델 가중치 로드
model = AutoModelForCausalLM.from_pretrained(model_name)

# 직접 로드한 model/tokenizer 객체를 pipeline에 주입해 생성 파이프라인 구성
kogpt2_gen = pipeline(
    'text-generation',
    model=model,
    tokenizer=tokenizer
)

prompt_ko = '인공지능 기술이 발전하면서'

result_ko = kogpt2_gen(
    prompt_ko,
    max_new_tokens=80,           # 새로 생성할 최대 토큰 수
    num_return_sequences=1,      # 결과 1개만 생성
    temperature=0.7,             # 다양성(0.7은 적당히 안정적)
    do_sample=True,              # 확률 샘플링 사용
    repetition_penalty=1.2       # 같은 표현 반복에 패널티를 줘 반복 방지
)

print('=== KoGPT2 한국어 생성 결과 ===')
print(result_ko[0]['generated_text'])
```

### (4) prompt 입력 → output 생성 → 반복 실험

좋은 텍스트를 생성하기 위해서는 프롬프트 설계(Prompt Engineering)가 매우 중요합니다. 같은 모델이라도 프롬프트를 어떻게 작성하느냐에 따라 결과의 품질이 크게 달라집니다.

```python
from transformers import pipeline

generator = pipeline('text-generation', model='gpt2')

# 프롬프트 유형별 비교 실험
# 같은 모델이라도 프롬프트 작성 방식에 따라 결과가 어떻게 달라지는지 비교합니다.
prompts = {
    '기본형': 'Write a story about a robot.',
    '상세 지시형': 'Write a short science fiction story about a robot who learns to feel emotions. The story should be 3 paragraphs.',
    '대화형': 'Human: Tell me about the future of AI.\nAssistant:',
    '계속 쓰기형': 'Once upon a time, in a world where AI governed all human decisions,',
}

# 모든 프롬프트에 공통으로 적용할 생성 옵션을 딕셔너리로 묶어 둡니다
gen_kwargs = dict(max_new_tokens=120, temperature=0.85, do_sample=True, pad_token_id=50256)

# 프롬프트 종류별로 순회하며 생성 → **gen_kwargs로 공통 옵션을 풀어서 전달
for style, prompt in prompts.items():
    result = generator(prompt, **gen_kwargs)[0]['generated_text']
    print(f"\n{'='*50}")     # 구분선 출력
    print(f'[{style}]')       # 프롬프트 유형 표시
    print(result[:300])  # 300자만 출력
```

*[코드 2-15] 프롬프트 유형별 비교 실험*

### (5) 생성 길이, 반복 방지, top-k, top-p 등의 파라미터 조정법

텍스트 생성 품질은 다양한 하이퍼파라미터(hyperparameter)에 의해 결정됩니다. 각 파라미터의 역할을 이해하고 적절히 조정하는 것이 고품질 텍스트 생성의 핵심입니다.

| 파라미터 | 기본값 | 역할 및 권장 범위 |
| --- | --- | --- |
| `max_new_tokens` | 20 | 새로 생성할 최대 토큰 수. 권장: 50~500 |
| `min_new_tokens` | 0 | 최소 생성 토큰 수. 너무 짧은 출력 방지에 유용 |
| `temperature` | 1.0 | 생성 다양성 조절. 낮을수록(0.1~0.5) 안정적, 높을수록(0.8~1.5) 창의적 |
| `do_sample` | False | True: 확률 샘플링, False: greedy decoding (항상 가장 높은 확률 선택) |
| `top_k` | 50 | 상위 k개 토큰 중에서만 샘플링. 작을수록 일관성 증가 (권장: 30~100) |
| `top_p` (nucleus) | 1.0 | 누적 확률 p 이하의 토큰에서만 샘플링 (권장: 0.8~0.95) |
| `repetition_penalty` | 1.0 | 반복 토큰에 패널티 부여. 1.0=비활성, 권장: 1.1~1.5 |
| `num_return_sequences` | 1 | 생성할 독립 시퀀스 수. 다양한 결과 비교 시 2~5 권장 |
| `no_repeat_ngram_size` | 0 | n-gram 반복 방지. 2~4 설정 시 반복 구절 차단 |

```python
from transformers import pipeline

generator = pipeline('text-generation', model='gpt2', pad_token_id=50256)
prompt = 'The best way to learn programming is'

# ── Greedy Decoding: 항상 가장 확률 높은 토큰 선택 (결정론적)
# do_sample=False → 매번 동일한 결과, 안전하지만 단조로울 수 있음
greedy = generator(prompt, max_new_tokens=50, do_sample=False)

# ── Beam Search: 상위 k개 경로를 동시에 탐색 (높은 품질)
# num_beams=5 → 5개 후보 경로를 유지하며 가장 점수 높은 문장 선택, early_stopping=True는 종료 조건 충족 시 조기 중단
beam = generator(prompt, max_new_tokens=50, num_beams=5, early_stopping=True)

# ── Top-k Sampling: 상위 50개 토큰에서 확률 샘플링 (다양성)
# top_k=50 → 확률 상위 50개 후보 중에서만 무작위 선택
topk = generator(prompt, max_new_tokens=50, do_sample=True, top_k=50, temperature=0.9)

# ── Nucleus (Top-p) Sampling: 누적 확률 0.92 이내 토큰에서 샘플링
# top_p=0.92 → 확률을 큰 순서로 더해 0.92가 될 때까지의 후보군에서만 선택
nucleus = generator(prompt, max_new_tokens=50, do_sample=True, top_p=0.92, temperature=0.9)

# ── 반복 방지 조합: repetition_penalty + no_repeat_ngram_size
no_repeat = generator(
    prompt,
    max_new_tokens=100,
    do_sample=True,
    temperature=0.8,
    repetition_penalty=1.3,    # 이미 나온 토큰을 다시 고를 확률을 낮춤
    no_repeat_ngram_size=3     # 동일한 3-gram(연속 3토큰)이 반복되지 않도록 차단
)

# 각 전략의 결과에서 프롬프트 부분([len(prompt):])을 잘라 내고 80자만 출력해 비교합니다
print('[Greedy] ', greedy[0]['generated_text'][len(prompt):][:80])
print('[Beam-5] ', beam[0]['generated_text'][len(prompt):][:80])
print('[Top-k]   ', topk[0]['generated_text'][len(prompt):][:80])
print('[Nucleus] ', nucleus[0]['generated_text'][len(prompt):][:80])
```

*[코드 2-16] 다양한 디코딩 전략 비교*

> 실무에서는 `temperature=0.7~0.9`, `top_p=0.9`, `repetition_penalty=1.2~1.3`의 조합이 창의적이면서도 일관성 있는 텍스트를 생성하는 데 효과적입니다.

### (3) AutoTokenizer, AutoModelForSequenceClassification 직접 사용하기

KoBERT를 이용한 한국어 감성 분석을 `AutoTokenizer`와 `AutoModelForSequenceClassification`으로 직접 구현해 봅니다. `pipeline()`을 사용하지 않고 하위 수준에서 직접 다루면 모델 동작을 더 세밀하게 제어할 수 있습니다.

```python
import torch
from transformers import AutoTokenizer, AutoModelForSequenceClassification

# ── 한국어 감성 분석 모델 로드 ─────────────────────────────
# snunlp/KR-FinBert-SC: 금융/일반 텍스트 감성 분석 파인튜닝 모델
model_name = 'snunlp/KR-FinBert-SC'

print(f'모델 로딩 중: {model_name}')
tokenizer = AutoTokenizer.from_pretrained(model_name)              # 한국어 토크나이저 로드

model.eval()                                                      # 추론 모드 전환

# id2label: 모델이 출력하는 클래스 번호(0,1,2)가 각각 어떤 감성인지 보여 줍니다
print(f'레이블 구성: {model.config.id2label}')
# 출력: {0: 'negative', 1: 'neutral', 2: 'positive'}
```

*[코드 2-17] 한국어 감성 분석 모델 로드*

```python
def predict_sentiment(text: str) -> dict:
    """한국어 텍스트의 감성을 분석하여 레이블과 점수를 반환합니다."""
    # 토크나이징: 텍스트를 모델 입력 텐서로 변환
    inputs = tokenizer(
        text,
        return_tensors='pt',  # PyTorch 텐서로 반환
        truncation=True,      # max_length 초과분은 잘라냄
        max_length=512,       # 최대 토큰 길이(BERT 계열 한계)
        padding=True          # 짧으면 패딩으로 길이 맞춤
    )

    # 추론: 그래디언트 계산 없이 forward만 실행
    with torch.no_grad():
        outputs = model(**inputs)
        logits = outputs.logits  # 클래스별 원시 점수

    # 소프트맥스 변환: logits를 확률로 바꾸고 [0]으로 첫(유일) 샘플 선택
    probs = torch.nn.functional.softmax(logits, dim=-1)[0]
    predicted_id = probs.argmax().item()              # 가장 높은 확률의 클래스 번호
    label = model.config.id2label[predicted_id]       # 번호를 레이블 문자열로 변환
    score = probs[predicted_id].item()                # 그 레이블의 확률값

    # 결과를 보기 좋은 딕셔너리로 구성해 반환
    return {
        'text': text,
        'label': label,
        'score': round(score, 4),
        'all_scores': {
            # 모든 클래스의 확률을 {레이블: 점수} 형태로 모음
            model.config.id2label[i]: round(p.item(), 4)
            for i, p in enumerate(probs)
        }
    }

# 테스트
test_sentences = [
    '오늘 주가가 급등하여 투자자들이 기뻐하고 있습니다.',
    '경기 침체 우려로 시장이 크게 하락했습니다.',
    '삼성전자가 새로운 반도체 공장 건설을 발표했습니다.',
]

# 각 문장을 분석해 레이블과 점수를 정렬된 형태로 출력
for sent in test_sentences:
    result = predict_sentiment(sent)
    print(f'[{result["label"]:8s} {result["score"]:.4f}] {result["text"]}')
```

### (4) pipeline 없이 모델 로딩해서 직접 inference 해보기

KoELECTRA를 이용한 개체명 인식(NER) 태스크를 pipeline 없이 직접 구현해 봅니다. NER은 텍스트에서 인물, 조직, 장소, 날짜 등의 개체를 식별하는 태스크입니다.

```python
import torch
# AutoModelForTokenClassification: 토큰마다 레이블을 붙이는 NER/품사 태깅용 모델 클래스
from transformers import AutoTokenizer, AutoModelForTokenClassification

model_name = 'monologg/koelectra-base-v3-naver-ner'  # 네이버 NER 데이터로 학습한 KoELECTRA
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForTokenClassification.from_pretrained(model_name)
model.eval()

def predict_ner(text: str) -> list:
    """텍스트에서 개체명을 인식합니다."""
    # 토크나이징 (offset_mapping으로 원래 텍스트 위치 추적)
    # return_offsets_mapping=True: 각 토큰이 원문에서 차지하는 (시작, 끝) 글자 위치를 함께 반환
    inputs = tokenizer(
        text,
        return_tensors='pt',
        return_offsets_mapping=True,
        truncation=True,
        max_length=512
    )
    offset_mapping = inputs.pop('offset_mapping')  # 모델 입력에는 불필요하므로 꺼내서 따로 보관

    # 토큰별 예측: 각 토큰 위치에서 가장 점수 높은 레이블 인덱스를 구함
    with torch.no_grad():
        outputs = model(**inputs)
        predictions = outputs.logits.argmax(dim=-1)[0]

    # 결과 파싱: 토큰 ID를 다시 토큰 문자열로 변환
    tokens = tokenizer.convert_ids_to_tokens(inputs['input_ids'][0])
    entities = []          # 최종 개체 목록
    current_entity = None  # 현재 이어 붙이는 중인 개체

    # 토큰 / 예측 레이블 / 원문 위치(start,end)를 동시에 순회
    for token, pred, (start, end) in zip(
        tokens, predictions, offset_mapping[0]
    ):
        label = model.config.id2label[pred.item()]  # 예측 번호 → 레이블 문자열(BIO 태그)
        if label == 'O' or start == end:  # 특수 토큰 또는 비-개체
            # 개체가 아닌 토큰을 만나면, 모으던 개체가 있으면 확정해 저장
            if current_entity:
                entities.append(current_entity)
                current_entity = None
        elif label.startswith('B-'):
            # B-(Begin): 새 개체의 시작 → 직전 개체를 닫고 새 개체 시작
            if current_entity:
                entities.append(current_entity)
            current_entity = {'entity': label[2:], 'word': text[start:end], 'start': start, 'end': end}
        elif label.startswith('I-') and current_entity:
            # I-(Inside): 진행 중인 개체의 연속 → 끝 위치를 늘려 단어를 확장
            current_entity['word'] = text[current_entity['start']:end]
            current_entity['end'] = end

    # 루프 종료 후 마지막으로 모으던 개체가 남아 있으면 저장
    if current_entity:
        entities.append(current_entity)

    return entities

# 테스트
text = '이순신 장군은 1545년 한성부에서 태어났으며 조선의 명장이었습니다.'
entities = predict_ner(text)
# 인식된 개체의 종류와 단어를 출력
for ent in entities:
    print(f' [{ent["entity"]:6s}] {ent["word"]}')
```

*[코드 2-19] KoELECTRA NER 직접 구현*

### (5) 사전 학습 모델과 파인튜닝된 모델의 차이 이해

Hugging Face Hub에는 두 종류의 모델이 공존합니다. 하나는 방대한 데이터로 일반 언어 능력을 학습한 사전 학습 모델(Pre-trained Model)이고, 다른 하나는 특정 태스크를 위해 추가 학습된 파인튜닝 모델(Fine-tuned Model)입니다.

| 구분 | 사전 학습 모델 | 파인튜닝 모델 |
| --- | --- | --- |
| 목적 | 일반 언어 능력 학습 | 특정 태스크 해결 |
| 학습 데이터 | 방대한 일반 텍스트 (수 TB) | 소량의 태스크 전용 레이블 데이터 |
| 출력 | 텍스트 표현(임베딩) 또는 마스크 예측 | 분류 레이블, 생성 텍스트 등 |
| 직접 추론 | 일반적으로 직접 사용 불가 (분류 헤드 없음) | 즉시 추론 가능 |
| 예시 (BERT) | `bert-base-uncased`, `monologg/koelectra-base-v3-discriminator` | `distilbert-base-uncased-finetuned-sst-2-english` |
| 활용 방법 | 임베딩 추출, 전이학습 베이스 | 즉시 pipeline() 적용 가능 |

> 파인튜닝 모델을 선택할 때는 Hub 모델 카드(Model Card)의 Evaluation Results를 반드시 확인하세요. 동일 태스크라도 학습에 사용된 데이터셋과 성능 지표가 모델마다 크게 다릅니다.

## 2.4 모델 저장과 불러오기 (로컬 & API)

### (1) 모델을 다운로드해서 오프라인에서 사용하기

인터넷 연결 없이 Hugging Face 모델을 사용하려면 모델을 미리 다운로드하여 로컬에 저장해야 합니다. 이는 운영 환경(프로덕션)에서의 안정성과 응답 속도를 위해 매우 중요합니다.

Hugging Face는 처음 `from_pretrained()` 호출 시 자동으로 로컬 캐시에 모델을 저장합니다. 기본 캐시 경로는 다음과 같습니다.

| 운영체제 | 기본 캐시 경로 |
| --- | --- |
| Linux / Mac | `~/.cache/huggingface/hub/` |
| Windows | `C:\Users\{사용자명}\.cache\huggingface\hub\` |
| 사용자 지정 | 환경변수 `HF_HOME` 또는 `TRANSFORMERS_CACHE` 설정으로 변경 가능 |

```python
# 방법 1: 환경변수로 캐시 경로 지정 (from_pretrained() 호출 전에 설정)
import os
# 다운로드된 모델이 저장될 기본 위치를 바꿉니다 (디스크 용량 관리에 유용)
os.environ['TRANSFORMERS_CACHE'] = '/data/models/huggingface'

# 방법 2: from_pretrained()의 cache_dir 파라미터로 직접 지정
from transformers import AutoTokenizer, AutoModelForSequenceClassification

custom_cache = './models/cache'
# 이 호출에 한해서만 지정한 폴더에 캐시하도록 cache_dir로 경로를 넘깁니다
tokenizer = AutoTokenizer.from_pretrained(
    'distilbert-base-uncased-finetuned-sst-2-english',
    cache_dir=custom_cache
)

# 방법 3: 오프라인 모드 강제 (로컬 캐시만 사용)
os.environ['TRANSFORMERS_OFFLINE'] = '1'  # '1'이면 네트워크 접근을 막음
# 이 설정 이후에는 인터넷 접근 없이 캐시된 모델만 로드됩니다
```

### (3) 로컬 캐싱된 모델 경로 구조 이해

Hugging Face의 기본 캐시 디렉토리 내부 구조를 이해하면 모델 파일을 효율적으로 관리할 수 있습니다.

```python
# 캐시 내 저장 구조 확인 코드
import os
from pathlib import Path  # 경로를 객체로 다루는 표준 라이브러리

# 사용자 홈 디렉토리 하위의 기본 Hugging Face 캐시 경로 구성
cache_dir = Path.home() / '.cache' / 'huggingface' / 'hub'

if cache_dir.exists():  # 캐시 폴더가 존재할 때만 실행
    print(f'캐시 디렉토리: {cache_dir}')
    total_size = 0
    # 캐시 폴더 안의 항목을 정렬해 순회
    for item in sorted(cache_dir.iterdir()):
        if item.is_dir():  # 모델별 하위 폴더만 대상
            # rglob('*')로 하위 모든 파일을 찾아 크기를 합산
            size = sum(f.stat().st_size for f in item.rglob('*') if f.is_file())
            total_size += size
            print(f' {item.name[:50]:50s} {size/1024/1024:.1f} MB')  # 모델별 용량(MB)
    # 전체 캐시 용량을 GB 단위로 출력
    print(f'\n총 캐시 용량: {total_size/1024/1024/1024:.2f} GB')

# 캐시 디렉토리 구조 예시:
# models--distilbert-base-uncased-finetuned-sst-2-english   264.3 MB
# models--skt--kogpt2-base-v2                                487.1 MB
# models--bert-base-uncased                                  420.2 MB
```

*[코드 2-23] 캐시 디렉토리 용량 확인*

> 모델 캐시가 너무 커지면 디스크 공간이 부족해질 수 있습니다. `huggingface-cli cache scan` 명령으로 캐시를 스캔하고, `huggingface-cli cache delete` 명령으로 불필요한 모델을 삭제할 수 있습니다.

### (5) Streamlit 또는 FastAPI로 저장된 모델 활용 실습

로컬에 저장된 모델을 Streamlit 웹 앱으로 빠르게 시연하는 코드를 작성합니다. Streamlit은 Python 코드만으로 대화형 웹 앱을 만들 수 있어 모델 데모에 매우 적합합니다.

```python
# app_streamlit.py
# 실행: streamlit run app_streamlit.py
# pip install streamlit transformers torch 필요

import streamlit as st          # 웹 UI를 파이썬만으로 만드는 라이브러리
from transformers import pipeline

# 페이지 메타 설정(브라우저 탭 제목 등)
st.set_page_config(page_title='감성 분석 데모', page_icon='')
st.title(' 한국어 감성 분석 데모')  # 페이지 큰 제목
st.markdown('**snunlp/KR-FinBert-SC** 모델을 사용한 실시간 감성 분석입니다.')  # 설명 텍스트(마크다운)

# 모델 캐싱: @st.cache_resource로 앱 재실행 시 재로드 방지
# Streamlit은 상호작용마다 스크립트를 다시 실행하므로, 이 데코레이터로 모델을 한 번만 메모리에 올립니다.
@st.cache_resource
def load_model():
    return pipeline(
        'text-classification',
        model='snunlp/KR-FinBert-SC',
        return_all_scores=True       # 모든 레이블 점수를 받아 막대로 표시하기 위함
    )

classifier = load_model()  # 캐시된 모델 사용(최초 1회만 실제 로드)

# 사용자 입력: 여러 줄 텍스트 입력 상자
user_input = st.text_area(
    '분석할 텍스트를 입력하세요:',
    placeholder='예) 오늘 주가가 급등하여 투자자들이 기뻐하고 있습니다.',  # 입력 예시
    height=120
)

# 버튼이 눌리고(and) 입력이 비어 있지 않을 때만 분석 실행
if st.button('감성 분석하기', type='primary') and user_input:
    with st.spinner('분석 중...'):          # 처리 중 로딩 스피너 표시
        results = classifier(user_input)[0]  # 입력 텍스트 분석, 첫 결과의 점수 리스트

    st.subheader('분석 결과')
    # 점수가 높은 레이블부터 정렬해 막대로 시각화
    for res in sorted(results, key=lambda x: x['score'], reverse=True):
        label = res['label']
        score = res['score']
        # 레이블에 따라 색상 결정(긍정=초록, 부정=빨강, 그 외=회색)
        color = 'green' if label == 'positive' else ('red' if label == 'negative' else 'gray')
        st.progress(score)  # 점수를 진행 막대로 표시
        # 레이블과 확신도를 퍼센트(%)로 출력
        st.markdown(f':{color}[**{label.upper()}**] — 확신도: {score:.2%}')
```

*[코드 2-25] Streamlit 감성 분석 데모 앱*

> `@st.cache_resource` 데코레이터는 Streamlit에서 매우 중요합니다. 이 데코레이터 없이는 사용자가 버튼을 클릭할 때마다 모델이 다시 로드되어 느린 응답 시간이 발생합니다. `cache_resource`를 사용하면 앱이 실행되는 동안 모델이 메모리에 한 번만 올라갑니다.

## 2.5 FastAPI로 감정 분석 API 만들기

FastAPI는 Python의 현대적인 웹 프레임워크로, 빠른 성능과 자동 문서화 기능을 제공합니다. Hugging Face 모델과 FastAPI를 결합하면 실무 환경에서 사용 가능한 AI API 서버를 빠르게 구축할 수 있습니다.

### FastAPI 선택 이유

- 비동기(async) 처리 지원으로 높은 동시 요청 처리 성능
- OpenAPI(Swagger) 문서 자동 생성 — `/docs` 엔드포인트 접속으로 즉시 확인
- Pydantic 기반 타입 검증으로 잘못된 입력 자동 처리
- Django, Flask 대비 2~5배 빠른 응답 속도 (Starlette 기반)
- 타입 힌팅(Type Hints)으로 IDE 자동완성 완벽 지원

### (1) FastAPI 라우터에서 transformer 모델을 불러오기

FastAPI 애플리케이션의 시작 시점에 모델을 한 번만 로드하고, 이후 모든 요청에서 재활용하는 패턴이 핵심입니다. lifespan 이벤트 핸들러를 사용하면 앱 시작 시 모델을 초기화하고 종료 시 자원을 해제할 수 있습니다.

```python
# main.py
# 디렉토리 구조:
# sentiment_api/
# ├── main.py          (이 파일)
# ├── models.py        (Pydantic 스키마)
# └── requirements.txt

from contextlib import asynccontextmanager  # 시작/종료 로직을 묶는 컨텍스트 매니저 데코레이터
from fastapi import FastAPI
from transformers import pipeline
import torch
import time

# ── 전역 모델 저장소 ──────────────────────────────────────
# 앱 시작 시 로드한 모델을 여기 담아 두고 모든 요청에서 재사용합니다(요청마다 재로드 방지).
ml_models = {}

@asynccontextmanager
async def lifespan(app: FastAPI):
    """앱 시작 시 모델 로드, 종료 시 메모리 해제"""
    # Startup: 모델 로드 (yield 이전 코드 = 서버 기동 시 1회 실행)
    print('모델 로딩 시작...')
    start = time.time()

    device = 0 if torch.cuda.is_available() else -1  # GPU 있으면 0, 없으면 CPU(-1)
    ml_models['sentiment'] = pipeline(
        'text-classification',
        model='snunlp/KR-FinBert-SC',
        device=device,
        return_all_scores=True
    )

    elapsed = time.time() - start                     # 로딩 소요 시간
    device_name = 'GPU' if device == 0 else 'CPU'
    print(f'모델 로드 완료 ({device_name}, {elapsed:.2f}초)')

    yield  # 앱 실행 중 (이 지점에서 서버가 요청을 처리)

    # Shutdown: 메모리 해제 (yield 이후 코드 = 서버 종료 시 실행)
    ml_models.clear()
    print('모델 메모리 해제 완료')

# FastAPI 앱 생성: title/description/version은 자동 생성되는 API 문서에 표시됨
app = FastAPI(
    title='한국어 감성 분석 API',
    description='KR-FinBERT 모델 기반 한국어 감성 분석 REST API',
    version='1.0.0',
    lifespan=lifespan  # 위에서 정의한 시작/종료 핸들러 등록
)
```

*[코드 2-26] FastAPI 앱 초기화 및 모델 로드 (lifespan 사용)*

### (2) 텍스트 입력 → 감정 분석 결과 리턴 API 구축

API 요청/응답 스키마를 Pydantic 모델로 정의하고, 엔드포인트 함수를 작성합니다.

```python
# models.py — Pydantic 스키마 정의
# Pydantic 모델은 요청/응답의 형태를 정의하고 입력 유효성 검사를 자동으로 수행합니다.
from pydantic import BaseModel, Field
from typing import Optional

# 요청 본문 스키마: 클라이언트가 보내야 하는 데이터 형식 정의
class SentimentRequest(BaseModel):
    text: str = Field(
        ...,                 # ...(Ellipsis)은 필수 필드를 의미
        min_length=1,        # 최소 1자
        max_length=512,      # 최대 512자(검증 실패 시 자동 422 오류)
        description='감성 분석할 한국어 텍스트 (1~512자)',
        example='오늘 주가가 급등하여 투자자들이 기뻐하고 있습니다.'  # 문서에 표시될 예시
    )
    include_all_scores: Optional[bool] = Field(
        default=False,       # 선택 필드, 기본값 False
        description='True이면 모든 레이블의 점수를 함께 반환'
    )

# 개별 점수 스키마: 레이블과 점수 한 쌍
class SentimentScore(BaseModel):
    label: str = Field(description='예측 레이블: positive, negative, neutral')
    score: float = Field(description='해당 레이블의 확신도 (0.0 ~ 1.0)')

# 응답 본문 스키마: 서버가 돌려줄 데이터 형식 정의
class SentimentResponse(BaseModel):
    text: str                                           # 입력 텍스트 echo
    prediction: SentimentScore                          # 최고 점수 예측 결과
    all_scores: Optional[list[SentimentScore]] = None   # 옵션: 전체 레이블 점수
    processing_time_ms: float                           # 처리 소요 시간(ms)
```

### (3) POST 방식으로 입력을 받아 처리하는 예제

```python
# main.py 에 추가할 엔드포인트 코드
from fastapi import HTTPException  # 오류 응답을 던질 때 사용
from models import SentimentRequest, SentimentResponse, SentimentScore  # 앞서 정의한 스키마

# @app.post: POST /api/v1/sentiment 경로로 들어오는 요청을 이 함수가 처리
@app.post(
    '/api/v1/sentiment',
    response_model=SentimentResponse,  # 응답을 이 스키마로 검증·직렬화
    summary='감성 분석',                # API 문서용 짧은 설명
    description='입력 텍스트의 긍정/부정/중립을 분석합니다.'
)
async def analyze_sentiment(request: SentimentRequest) -> SentimentResponse:
    """
    ## 감성 분석 API
    - **text**: 분석할 한국어 텍스트 (필수)
    - **include_all_scores**: 모든 레이블 점수 포함 여부
    """
    # 모델이 아직 로드되지 않았으면 503(서비스 불가) 반환
    if 'sentiment' not in ml_models:
        raise HTTPException(status_code=503, detail='모델이 아직 로드되지 않았습니다.')

    start = time.time()  # 처리 시간 측정 시작
    try:
        # 전역 저장소의 모델로 추론, [0]으로 첫 입력의 점수 리스트 추출
        raw_results = ml_models['sentiment'](request.text)[0]
    except Exception as e:
        # 추론 중 예외 발생 시 500(서버 오류)로 변환
        raise HTTPException(status_code=500, detail=f'추론 오류: {str(e)}')
    elapsed_ms = (time.time() - start) * 1000  # 경과 시간을 ms로 환산

    # 최고 점수 레이블 추출: score가 가장 큰 항목을 prediction으로 사용
    best = max(raw_results, key=lambda x: x['score'])
    prediction = SentimentScore(label=best['label'], score=round(best['score'], 4))

    # 전체 점수 (옵션): include_all_scores=True일 때만 모든 레이블을 점수 내림차순으로 구성
    all_scores = None
    if request.include_all_scores:
        all_scores = [
            SentimentScore(label=r['label'], score=round(r['score'], 4))
            for r in sorted(raw_results, key=lambda x: x['score'], reverse=True)
        ]

    # 응답 스키마에 맞춰 결과 반환 (자동으로 JSON 직렬화됨)
    return SentimentResponse(
        text=request.text,
        prediction=prediction,
        all_scores=all_scores,
        processing_time_ms=round(elapsed_ms, 2)
    )

# 헬스체크 엔드포인트: GET /health 로 서버/모델 상태를 간단히 확인
@app.get('/health', summary='서버 상태 확인')
async def health_check():
    return {
        'status': 'healthy',                          # 서버 동작 여부
        'model_loaded': 'sentiment' in ml_models,     # 모델 로드 여부
        'device': 'GPU' if torch.cuda.is_available() else 'CPU'  # 실행 장치
    }
```

### (5) 실시간 예측 응답 속도 측정 및 개선

프로덕션 환경에서는 응답 속도(Latency)와 처리량(Throughput)이 매우 중요합니다. 다음은 API 성능을 측정하고 개선하는 핵심 기법들입니다.

| 최적화 기법 | 효과 | 구현 방법 |
| --- | --- | --- |
| 배치 처리 | 처리량 2~10배 향상 | 여러 요청을 모아 한 번에 모델에 전달 |
| 모델 양자화 (INT8) | 메모리 50% 감소, 속도 1.5~3배 향상 | `torch.quantization` 또는 bitsandbytes 사용 |
| fp16/bf16 변환 | 메모리 50% 감소, GPU 속도 향상 | `torch_dtype=torch.float16` 설정 |
| ONNX 변환 | CPU 추론 2~5배 향상 | optimum 라이브러리로 ONNX 내보내기 |
| 모델 캐싱 | 재로드 시간 제거 | `@st.cache_resource` 또는 lifespan 이벤트 사용 |
| 비동기 추론 | 대기 시간 감소 | `asyncio` + `ThreadPoolExecutor` 조합 |

```python
# ── 배치 처리로 처리량 향상 ───────────────────────────────
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List

# 배치 요청 스키마: 여러 텍스트를 리스트로 한 번에 받음
class BatchRequest(BaseModel):
    texts: List[str]  # 최대 32개 텍스트 동시 처리

@app.post('/api/v1/sentiment/batch')
async def batch_sentiment(request: BatchRequest):
    # 과부하 방지를 위해 한 번에 처리할 수 있는 개수를 32개로 제한
    if len(request.texts) > 32:
        raise HTTPException(status_code=400, detail='최대 32개까지 처리 가능합니다.')

    start = time.time()
    # 배치 전체를 한 번의 모델 호출로 처리 (개별 호출보다 훨씬 효율적)
    all_results = ml_models['sentiment'](request.texts)
    elapsed_ms = (time.time() - start) * 1000

    return {
        # 입력 텍스트와 결과를 짝지어, 각 결과의 최고 점수 레이블을 prediction으로 담음
        'results': [
            {'text': t, 'prediction': max(r, key=lambda x: x['score'])}
            for t, r in zip(request.texts, all_results)
        ],
        'total_texts': len(request.texts),                          # 처리한 텍스트 수
        'total_time_ms': round(elapsed_ms, 2),                      # 전체 처리 시간
        'avg_time_ms': round(elapsed_ms / len(request.texts), 2)    # 텍스트당 평균 시간
    }
```

```python
# ── fp16 양자화로 GPU 성능 향상 ──────────────────────────
import torch
from transformers import pipeline

# fp16 적용 (GPU 환경에서만 유효)
# float16(반정밀도)은 GPU 텐서 코어에서 빠르게 계산되며 메모리를 절반만 사용합니다.
if torch.cuda.is_available():
    fp16_pipe = pipeline(
        'text-classification',
        model='snunlp/KR-FinBert-SC',
        device=0,
        torch_dtype=torch.float16  # 메모리 절반, 속도 향상
    )

# ── 응답 시간 벤치마크 ────────────────────────────────────
import statistics  # 평균·중앙값 등 통계 계산용 표준 라이브러리

def benchmark(pipe, texts, n_runs=50):
    times = []  # 각 실행의 소요 시간(ms)을 모을 리스트
    # 동일 입력으로 n_runs번 반복 실행해 시간 분포를 수집
    for _ in range(n_runs):
        start = time.time()
        pipe(texts)                                  # 추론 실행
        times.append((time.time() - start) * 1000)   # 경과 시간(ms) 기록

    # 수집한 시간들로 주요 지표 출력
    print(f'N={n_runs}회, 배치크기={len(texts)}')
    print(f' 평균: {statistics.mean(times):.2f} ms')      # 평균 응답 시간
    print(f' 중앙값: {statistics.median(times):.2f} ms')   # 중앙값(이상치 영향 적음)
    print(f' p95: {sorted(times)[int(n_runs*0.95)]:.2f} ms')  # 95퍼센타일(상위 5% 지연)
    print(f' 최솟값: {min(times):.2f} ms')
    print(f' 최댓값: {max(times):.2f} ms')
```
