## 8.1 FastAPI 운영 환경: Uvicorn과 Gunicorn

### (1) FastAPI 개발 vs 운영 환경 차이

FastAPI 개발 서버(`uvicorn --reload`)는 코드 변경을 자동 감지하고 재시작하는 편의 기능이 있지만, 단일 프로세스로만 동작하고 에러 처리가 느슨하여 실제 트래픽을 감당하기에 적합하지 않습니다. 운영 환경에서는 다중 워커(Multi-Worker) 프로세스, 구조화된 로깅, 안정적인 에러 처리, 보안 설정이 필수입니다.

| 구분 | 개발 환경 (Development) | 운영 환경 (Production) |
| --- | --- | --- |
| 실행 명령 | `uvicorn main:app --reload` | `gunicorn main:app -w 4 -k uvicorn.workers.UvicornWorker` |
| 워커 수 | 1개 (싱글 프로세스) | CPU 코어 × 2 + 1 (멀티 프로세스) |
| 자동 재시작 | 코드 변경 시 자동 재시작 | 크래시 발생 시에만 재시작 |
| 디버그 모드 | `debug=True`, 상세 오류 노출 | `debug=False`, 오류 은폐 |
| 로깅 | 콘솔 출력 (stdout) | 파일 로테이션 + 구조화 JSON 로그 |
| 보안 | CORS 전체 허용, HTTP | CORS 제한, HTTPS (Nginx), 인증 |
| 성능 | 낮음 (단일, 오버헤드 있음) | 높음 (멀티워커, 최적화) |

개발 환경과 운영 환경의 설정을 코드 레벨에서도 분리하는 것이 중요합니다. python-dotenv와 Pydantic Settings를 활용하면 환경변수 기반으로 설정을 손쉽게 관리할 수 있습니다.

```python
# config.py — Pydantic Settings로 환경별 설정 관리
from pydantic_settings import BaseSettings
from functools import lru_cache
from typing import Literal


# BaseSettings를 상속하면 클래스에 정의한 필드가 자동으로 환경변수/.env에서 채워집니다.
class Settings(BaseSettings):
    # 앱 기본 설정
    APP_NAME: str = "AI API Service"            # 애플리케이션 이름(로그/문서 표시용)
    APP_VERSION: str = "1.0.0"                  # 버전 문자열
    # Literal로 허용값을 제한 → 오타나 잘못된 환경값이 들어오면 검증 단계에서 에러 발생
    ENVIRONMENT: Literal["development", "staging", "production"] = "development"
    DEBUG: bool = False                         # 디버그 모드(운영에서는 반드시 False)

    # 서버 설정
    HOST: str = "0.0.0.0"                        # 0.0.0.0 = 모든 네트워크 인터페이스에서 수신
    PORT: int = 8000                            # 서비스 포트
    WORKERS: int = 1                            # 워커 프로세스 수(운영에서는 늘림)

    # 모델 설정
    MODEL_ID: str = "runwayml/stable-diffusion-v1-5"  # 사용할 HF 모델 ID
    MODEL_CACHE_DIR: str = "./model_cache"      # 모델 가중치 캐시 경로
    HF_TOKEN: str = ""                          # 비공개 모델 접근용 Hugging Face 토큰

    # 보안 설정
    SECRET_KEY: str = "change-this-in-production"  # 세션/서명용 비밀 키(운영에서 반드시 교체)
    ALLOWED_ORIGINS: list = ["*"]               # CORS 허용 출처(운영에서는 도메인 제한)
    API_KEY: str = ""                           # API 인증 키

    # 로깅 설정
    LOG_LEVEL: str = "INFO"                      # 로그 레벨(DEBUG/INFO/WARNING 등)
    LOG_FILE: str = "./logs/app.log"            # 로그 파일 경로

    class Config:
        env_file = ".env"                       # 어떤 .env 파일을 읽을지 지정
        env_file_encoding = "utf-8"             # 한글 주석 등 대비 인코딩 지정

    @property
    def is_production(self) -> bool:
        # 환경이 production인지 한 줄로 판별(분기 처리에 사용)
        return self.ENVIRONMENT == "production"

    @property
    def recommended_workers(self) -> int:
        # 권장 워커 수 계산식: CPU 코어 수 × 2 + 1 (일반 API 기준 경험칙)
        import os
        return (os.cpu_count() or 1) * 2 + 1


# lru_cache로 감싸 Settings를 한 번만 생성하고 재사용(매 호출마다 .env를 다시 읽지 않음)
@lru_cache()
def get_settings() -> Settings:
    return Settings()


# .env.development 예시
# ENVIRONMENT=development
# DEBUG=True
# LOG_LEVEL=DEBUG
# WORKERS=1


# .env.production 예시
# ENVIRONMENT=production
# DEBUG=False
# LOG_LEVEL=INFO
# WORKERS=5
# SECRET_KEY=supersecretkeyhere
# HF_TOKEN=hf_xxxxxxxxxxxx
# ALLOWED_ORIGINS=["https://myapp.com","https://www.myapp.com"]
```

### (2) Uvicorn 단독 실행의 한계와 Gunicorn 연동 이유

Uvicorn은 ASGI(Asynchronous Server Gateway Interface) 서버로, 단일 이벤트 루프에서 비동기 요청을 처리합니다. 비동기 I/O 작업에서는 매우 효율적이지만, CPU를 집중적으로 사용하는 AI 모델 추론에서는 싱글 프로세스가 병목이 됩니다. Gunicorn은 이 문제를 해결해주는 프리포크(pre-fork) 워커 매니저입니다.

#### Uvicorn 단독의 한계

1. **단일 프로세스 제약**: GIL(Global Interpreter Lock)으로 인해 CPU 바운드 작업이 다른 요청을 블로킹
2. **크래시 자동 복구 없음**: 워커 프로세스가 비정상 종료되면 서비스 전체 중단
3. **부드러운 재시작(Graceful Reload) 미지원**: 무중단 배포 불가
4. **CPU 코어 활용 불가**: 4코어 서버에서도 1코어만 사용

#### Gunicorn + UvicornWorker 조합의 장점

1. **멀티 프로세스**: 여러 Uvicorn 워커를 독립 프로세스로 실행 → 병렬 요청 처리
2. **자동 워커 재시작**: 워커 크래시 시 Gunicorn이 자동으로 새 워커 생성
3. **무중단 재시작**: `gunicorn --reload` 또는 SIGHUP 시그널로 순차 재시작
4. **워커 수 동적 조정**: 트래픽에 따라 `max_requests`, `timeout` 등 세밀한 조정 가능
5. **프로세스 격리**: 한 워커의 메모리 누수가 다른 워커에 영향 없음

```bash
# ── Uvicorn 단독 실행 (개발 전용) ───────────────────────
# main:app = main.py 파일의 app 객체. --reload는 코드 변경 시 자동 재시작(개발 편의)
# 단일 프로세스라 트래픽 처리량이 낮아 운영에는 부적합
uvicorn main:app --reload --host 0.0.0.0 --port 8000


# ── Gunicorn + UvicornWorker (운영 권장) ────────────────
# Gunicorn(프로세스 매니저)이 여러 Uvicorn 워커를 띄워 병렬 처리
gunicorn main:app \
  --worker-class uvicorn.workers.UvicornWorker \
  --workers 4 \
  --bind 0.0.0.0:8000   # --worker-class: ASGI 비동기 워커 / --workers: 워커 수 / --bind: 수신 주소:포트


# ── Gunicorn 워커 수 자동 계산 ──────────────────────────
# 일반 권장식: CPU 코어 수 × 2 + 1 (I/O 대기와 CPU 작업을 함께 고려한 경험칙)
WORKERS=$(python3 -c "import os; print(os.cpu_count() * 2 + 1)")
echo "권장 워커 수: $WORKERS"


# AI 모델 서비스 특화: 워커마다 모델이 메모리에 올라가므로 보수적으로 설정
# GPU 메모리가 제한적이면 워커 수를 줄여야 함(워커 수 × 모델 VRAM ≤ 전체 VRAM)
GPU_VRAM_GB=16        # 서버 전체 GPU 메모리(GB)
MODEL_VRAM_GB=4       # 모델 1개가 차지하는 VRAM(GB)
MAX_WORKERS=$(($GPU_VRAM_GB / $MODEL_VRAM_GB))   # VRAM을 넘지 않는 최대 워커 수
echo "VRAM 기반 최대 워커 수: $MAX_WORKERS"


# ── 완전한 운영 실행 명령어 ─────────────────────────────
gunicorn main:app \
  --worker-class uvicorn.workers.UvicornWorker \
  --workers 4 \
  --bind 0.0.0.0:8000 \
  --timeout 120 \              # 워커가 응답 없이 120초 넘으면 강제 재시작(AI 추론 시간 고려해 넉넉히)
  --keep-alive 5 \            # 클라이언트 Keep-Alive 연결 유지 시간(초)
  --max-requests 1000 \       # 워커가 1000요청 처리 후 재생성 → 메모리 누수 방지
  --max-requests-jitter 100 \ # 재생성 시점을 랜덤으로 분산해 동시 재시작 방지
  --graceful-timeout 30 \     # 종료 신호 후 진행 중 요청을 마칠 때까지 대기(초)
  --access-logfile ./logs/access.log \   # 접근 로그 파일
  --error-logfile ./logs/error.log \     # 에러 로그 파일
  --log-level info \          # 로그 상세도
  --pid ./gunicorn.pid        # 마스터 프로세스 PID 파일(재시작/종료 신호 전달용)
```

### (3) 운영 환경 실행 명령어 및 설정 최적화

Gunicorn 설정을 명령행 인자로 반복 입력하는 대신, `gunicorn.conf.py` 설정 파일로 관리하면 훨씬 체계적이고 버전 관리가 편리합니다.

```python
# gunicorn.conf.py — Gunicorn 운영 설정 파일
import os
import multiprocessing


# ── 서버 소켓 ────────────────────────────────────────────
bind = "0.0.0.0:8000"   # 수신 주소:포트(0.0.0.0 = 외부 접속 허용)
backlog = 2048    # 아직 수락(accept)되지 않고 대기하는 연결 큐 크기 (기본 2048)


# ── 워커 설정 ────────────────────────────────────────────
# AI 서비스: GPU 메모리 고려하여 워커 수 제한
# 일반 API: cpu_count * 2 + 1
cpu_count = multiprocessing.cpu_count()   # 현재 머신의 CPU 코어 수
# 환경변수 GUNICORN_WORKERS가 있으면 그 값, 없으면 (코어×2+1)과 4 중 작은 값(상한 4)
workers = int(os.getenv("GUNICORN_WORKERS", min(cpu_count * 2 + 1, 4)))

worker_class = "uvicorn.workers.UvicornWorker"   # ASGI 비동기 워커(FastAPI용)
worker_connections = 1000  # 비동기 워커 1개가 동시에 처리할 수 있는 연결 수

# 워커 수명 제한: 메모리 누수 방지 (1000요청 후 자동 재생성)
max_requests = 1000
max_requests_jitter = 100  # 재생성 시점을 ±100 범위로 흩어 동시 재시작 방지


# ── 타임아웃 설정 ────────────────────────────────────────
# AI 추론은 시간이 오래 걸릴 수 있으므로 넉넉히 설정
timeout = 120            # 워커가 응답 없이 이 시간을 넘기면 강제 재시작(초)
graceful_timeout = 30    # 종료 신호 후 진행 중 요청을 마칠 때까지 대기(초)
keepalive = 5            # Keep-Alive 연결 유지 시간 (초)


# ── 로깅 설정 ────────────────────────────────────────────
import logging
errorlog = "./logs/gunicorn_error.log"     # 에러 로그 파일 경로
accesslog = "./logs/gunicorn_access.log"   # 접근 로그 파일 경로
loglevel = os.getenv("LOG_LEVEL", "info").lower()   # 로그 레벨(환경변수 우선)

# 액세스 로그 포맷: JSON 구조화 로그 → Loki/ELK 등 로그 수집 도구와 연동 쉬움
# %(t)s 시각, %(h)s 클라이언트IP, %(m)s 메서드, %(U)s 경로, %(s)s 상태코드,
# %(b)s 응답바이트, %(D)s 처리시간(μs), %(f)s referer
access_log_format = ('{"time":"%(t)s","remote_ip":"%(h)s",'
                     '"method":"%(m)s","url":"%(U)s",'
                     '"status":%(s)s,"bytes":%(b)s,'
                     '"duration_ms":%(D)s,"referer":"%(f)s"}')


# ── 프로세스 관리 ────────────────────────────────────────
pidfile = "./gunicorn.pid"   # 마스터 PID 파일(재시작/종료 신호 전달용)
daemon = False     # True이면 백그라운드 데몬으로 실행(systemd 사용 시 False 권장)


# ── 보안 설정 ────────────────────────────────────────────
forwarded_allow_ips = "*"     # X-Forwarded-* 헤더를 신뢰할 프록시 IP(Nginx 뒤일 때 * 허용)
proxy_protocol = False        # PROXY 프로토콜 사용 여부


# ── 워커 프로세스 초기화 훅 ──────────────────────────────
def on_starting(server):
    # Gunicorn 마스터 시작 직전 1회 실행 → 로그 디렉터리 미리 생성
    import os
    os.makedirs("./logs", exist_ok=True)
    print(f"Gunicorn 시작: {workers} workers, timeout={timeout}s")


def worker_init(arbiter, worker):
    """각 워커 프로세스 시작 시 실행 (GPU 초기화·모델 사전 로드 등에 활용)"""
    print(f"워커 {worker.pid} 시작")


def on_exit(server):
    # Gunicorn 마스터 종료 시 1회 실행(정리 작업용)
    print("Gunicorn 종료 완료")


# ── 실행 명령어 (설정 파일 사용) ─────────────────────────
# gunicorn main:app -c gunicorn.conf.py
```

```ini
# ── systemd 서비스로 등록 (리눅스 서버) ──────────────────
# /etc/systemd/system/ai-api.service  (서버 부팅 시 자동 시작·죽으면 자동 재시작)
[Unit]

After=network.target          # 네트워크가 준비된 뒤에 서비스 시작

[Service]

WorkingDirectory=/home/ubuntu/ai-api          # 작업 디렉터리(코드 위치)
# 가상환경의 gunicorn으로 -c 설정 파일을 적용해 실행
ExecStart=/home/ubuntu/.venv/bin/gunicorn main:app -c gunicorn.conf.py
Restart=always                # 프로세스가 죽으면 항상 재시작
RestartSec=3                  # 재시작 전 3초 대기
Environment="PATH=/home/ubuntu/.venv/bin"     # 가상환경 실행 파일 경로 주입

[Install]
WantedBy=multi-user.target    # 일반 다중 사용자 부팅 단계에서 활성화
```

## 8.2 Hugging Face Spaces vs 자체 서버

### (1) Hugging Face Spaces의 장점과 한계

Hugging Face Spaces는 AI 모델 데모와 애플리케이션을 무료 또는 저비용으로 배포할 수 있는 호스팅 플랫폼입니다. Gradio 또는 Streamlit 앱을 저장소에 올리기만 하면 자동으로 빌드되고 공개 URL이 생성됩니다. 연구자, 학생, AI 애플리케이션 개발자들이 모델을 시연하고 공유하는 데 최적화되어 있습니다.

```python
# app.py — Gradio 기반 이미지 설명 AI 앱 (Space용)
import gradio as gr
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from PIL import Image


# ── 모델 로드 (Space 빌드 시 자동 실행) ─────────────────
model_id = "vikhyatk/moondream2"   # 경량 멀티모달(이미지+텍스트) 모델 ID

# 토크나이저 로드. revision으로 모델 버전을 고정해 재현성 확보
tokenizer = AutoTokenizer.from_pretrained(model_id, revision="2024-08-26")
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    trust_remote_code=True,       # 모델 저장소의 커스텀 코드 실행 허용(moondream에 필요)
    revision="2024-08-26",        # 버전 고정
    # GPU가 있으면 fp16(절반 정밀도, 빠름·메모리 절약), 없으면 fp32
    torch_dtype=torch.float16 if torch.cuda.is_available() else torch.float32
)
model.eval()   # 추론 모드로 전환(드롭아웃 등 학습 전용 동작 비활성화)


# ── 추론 함수 ─────────────────────────────────────────────
def describe_image(image: Image.Image, question: str) -> str:
    if image is None:
        return "이미지를 업로드해주세요."        # 이미지 없으면 안내 후 종료
    if not question.strip():
        question = "이 이미지에 대해 설명해주세요."  # 질문이 비면 기본 질문 사용
    enc = model.encode_image(image)              # 이미지를 모델 임베딩으로 인코딩
    return model.answer_question(enc, question, tokenizer)  # 질문에 대한 답변 생성


# ── Gradio UI ─────────────────────────────────────────────
# gr.Blocks: 행/열을 자유롭게 배치하는 유연한 레이아웃 컨테이너
with gr.Blocks(title="AI 이미지 설명기") as demo:
    gr.Markdown("## 🤖 AI 이미지 설명기")                        # 제목
    gr.Markdown("이미지를 업로드하고 질문을 입력하면 AI가 답변합니다.")  # 설명
    with gr.Row():                                   # 가로 배치
        with gr.Column():                            # 왼쪽 열: 입력
            img_input = gr.Image(type="pil", label="이미지 업로드")  # PIL 이미지로 받음
            q_input   = gr.Textbox(label="질문", value="이 이미지를 설명해주세요.")
            btn       = gr.Button("분석", variant="primary")        # 실행 버튼
        with gr.Column():                            # 오른쪽 열: 출력
            output    = gr.Textbox(label="AI 답변", lines=6)
    # 버튼 클릭 → describe_image 호출(입력 2개를 받아 output에 표시)
    btn.click(describe_image, inputs=[img_input, q_input], outputs=output)
    gr.Examples(                                     # 클릭으로 채워지는 예시 입력
        examples=[["./sample1.jpg", "이미지의 주요 내용은?"],
                  ["./sample2.jpg", "분위기를 설명해주세요."]],
        inputs=[img_input, q_input]
    )

demo.launch()   # 웹 서버 기동(Space에서는 자동으로 공개 URL 연결)
```

### (4)(5) 자체 서버 배포 대비 비교 및 선택 기준

프로젝트의 목적, 규모, 예산에 따라 최적의 배포 전략이 달라집니다. 학습/시연용과 실서비스 사이의 선택 기준을 명확히 이해하는 것이 중요합니다.

#### 배포 플랫폼 선택 기준 체크리스트

**Hugging Face Spaces를 선택해야 할 때:**

- 개인 포트폴리오, 연구 데모, 강의/세미나 시연 목적
- 예산이 거의 없는 학생/연구자
- HF Hub 모델을 그대로 사용하는 경우
- 빠른 프로토타입 배포 (30분 내 완성)

**자체 서버(AWS/GCP/Render 등)를 선택해야 할 때:**

- 실제 사용자 대상 프로덕션 서비스
- 지속적인 API 응답이 필요한 경우 (슬립 불가)
- 사용자 데이터 프라이버시/보안이 중요한 경우
- 커스텀 도메인, HTTPS, 인증 등 세밀한 제어 필요
- 모델을 직접 파인튜닝하여 운영하는 경우

## 8.3 Docker로 API 감싸기

### (1) Docker 기본 개념과 AI 서비스에서의 필요성

Docker는 애플리케이션과 그 실행 환경을 하나의 컨테이너(Container)로 패키징하는 플랫폼입니다. "내 노트북에서는 되는데 서버에서는 안 돼요" 문제를 완전히 해결합니다. AI 서비스에서 Docker가 특히 중요한 이유는 Python 패키지 버전, CUDA 버전, 시스템 라이브러리 등 의존성이 복잡하기 때문입니다.

| Docker 핵심 개념 | 설명 | 비유 |
| --- | --- | --- |
| Image (이미지) | 컨테이너의 설계도/스냅샷. 읽기 전용 레이어 구조 | 요리 레시피 |
| Container (컨테이너) | 이미지를 실행한 인스턴스. 격리된 실행 환경 | 레시피로 만든 요리 |
| Dockerfile | 이미지를 빌드하는 스크립트 파일 | 레시피 문서 |
| Registry (레지스트리) | 이미지를 저장/공유하는 저장소. Docker Hub, ECR 등 | 레시피 공유 사이트 |
| Docker Compose | 여러 컨테이너를 정의하고 함께 실행하는 도구 | 풀코스 요리 메뉴 |
| Volume (볼륨) | 컨테이너 외부에 데이터를 영구 저장하는 방법 | 냉장고 (데이터 보관) |
| Network (네트워크) | 컨테이너 간 통신을 위한 가상 네트워크 | 주방 내 통신망 |

```bash
# ── Docker 설치 확인 ─────────────────────────────────────
docker --version
# 출력: Docker version 25.0.3, build 4debf41

docker compose version
# 출력: Docker Compose version v2.24.6


# ── Docker 기본 명령어 정리 ──────────────────────────────
# 이미지 빌드 (-t: 이름:태그 지정, 끝의 . = 현재 폴더의 Dockerfile 사용)
docker build -t my-ai-api:latest .

# 컨테이너 실행 (-d: 백그라운드, --name: 이름, -p 호스트:컨테이너 포트 매핑)
docker run -d --name ai-api -p 8000:8000 my-ai-api:latest

# GPU 사용 컨테이너 실행 (--gpus all: 호스트의 모든 GPU 연결)
docker run -d --name ai-api --gpus all -p 8000:8000 my-ai-api:latest

# 실행 중인 컨테이너 목록
docker ps

# 컨테이너 로그 확인 (-f: 실시간 추적)
docker logs -f ai-api

# 컨테이너 내부 셸 접속 (-it: 대화형 터미널)
docker exec -it ai-api /bin/bash

# 컨테이너 중지 후 삭제
docker stop ai-api && docker rm ai-api

# 이미지 삭제
docker rmi my-ai-api:latest

# Docker Hub에 이미지 푸시 (tag로 레지스트리 경로를 붙인 뒤 push)
docker tag my-ai-api:latest myusername/my-ai-api:latest
docker push myusername/my-ai-api:latest
```

### (2) Dockerfile: FastAPI + 모델 환경 이미지 생성

AI 서비스를 위한 Dockerfile 작성에는 여러 주요 고려사항이 있습니다. CUDA 버전 선택, 패키지 레이어 캐싱 최적화, 불필요한 파일 제외, 보안 설정 등을 체계적으로 작성해야 합니다.

```dockerfile
# Dockerfile — FastAPI AI 서비스 이미지

# ── 베이스 이미지 선택 ───────────────────────────────────
# CUDA 필요 없는 경우 (CPU 전용)
# FROM python:3.11-slim
```

### (3) 모델 파일 포함 여부에 따른 이미지 최적화

AI 서비스 Docker 이미지의 가장 큰 문제는 모델 파일의 크기입니다. Stable Diffusion은 4~10GB, LLaVA-7B는 14GB를 초과합니다. 이 모델 파일을 Docker 이미지에 포함할지 외부에서 로드할지는 배포 전략에 큰 영향을 미칩니다.

| 전략 | 모델 포함 여부 | 이미지 크기 | 컨테이너 시작 시간 | 권장 상황 |
| --- | --- | --- | --- | --- |
| 전략 A: 이미지 내 포함 | O (이미지에 번들) | 매우 큼 (10~20GB) | 즉시 (다운로드 불필요) | 오프라인 환경, 엣지 배포 |
| 전략 B: HF Hub 다운로드 | X (런타임 다운로드) | 작음 (~2GB) | 느림 (첫 기동 시 수 GB 다운) | 클라우드, HF Hub 접근 가능 |
| 전략 C: Docker 볼륨 마운트 | X (외부 볼륨) | 작음 (~2GB) | 빠름 (로컬 파일) | 프로덕션 서버, 모델 공유 |
| 전략 D: S3/스토리지 다운로드 | X (S3에서 다운) | 작음 (~2GB) | 빠름 (내부 네트워크) | AWS 배포, 프라이빗 모델 |

### (4) Docker Compose와 Gunicorn 연동

Docker Compose는 여러 컨테이너를 하나의 YAML 파일(`docker-compose.yml`)로 정의하고 함께 실행하는 도구입니다. FastAPI 앱 + Nginx 리버스 프록시 + 모니터링 도구를 한 번에 구성하는 데 유용합니다.

```yaml
# docker-compose.yml — FastAPI + Nginx + Prometheus 통합 구성
version: "3.9"   # Compose 파일 포맷 버전

services:
  # ── FastAPI AI 앱 ────────────────────────────────────
  api:
    build:
      context: .              # 빌드 컨텍스트(현재 폴더)
      dockerfile: Dockerfile  # 사용할 Dockerfile
    image: ai-fastapi:latest  # 빌드 결과 이미지 태그
    container_name: ai_api
    restart: unless-stopped   # 수동 중지 외에는 항상 자동 재시작
    environment:              # 컨테이너 환경변수
      - ENVIRONMENT=production
      - HF_TOKEN=${HF_TOKEN}              # 호스트의 .env 값을 주입
      - MODEL_CACHE_DIR=/app/model_cache
      - WORKERS=2
    volumes:
      - model_cache:/app/model_cache     # 모델 캐시 영속화(named volume)
      - ./logs:/app/logs                 # 로그를 호스트에 보존
    expose:
      - "8000"           # 같은 네트워크 내부에만 공개(외부는 Nginx 경유)
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia   # GPU 1장을 컨테이너에 예약 할당
              count: 1
              capabilities: [gpu]
    healthcheck:               # 컨테이너 자체 헬스체크
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s        # 기동 직후 60초는 실패해도 무시(모델 로딩 시간)
    networks:
      - ai_network

  # ── Nginx 리버스 프록시 ──────────────────────────────
  nginx:
    image: nginx:1.25-alpine   # 경량 alpine 기반 Nginx
    container_name: nginx
    restart: unless-stopped
    ports:

      - "443:443"              # HTTPS를 외부에 노출
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro   # 설정 파일(읽기전용)
      - ./nginx/ssl:/etc/nginx/ssl:ro                 # TLS 인증서(읽기전용)
      - ./logs/nginx:/var/log/nginx                   # Nginx 로그 보존
    depends_on:
      api:
        condition: service_healthy   # api가 healthy가 된 뒤 시작
    networks:
      - ai_network

  # ── Prometheus 메트릭 수집 ───────────────────────────
  prometheus:
    image: prom/prometheus:latest
    container_name: prometheus
    restart: unless-stopped
    ports:

    volumes:
      - ./monitoring/prometheus.yml:/etc/prometheus/prometheus.yml:ro  # 스크랩 설정
      - prometheus_data:/prometheus    # 시계열 데이터 영속화
    command:
      - "--config.file=/etc/prometheus/prometheus.yml"
      - "--storage.tsdb.retention.time=30d"   # 메트릭 30일 보관
    networks:
      - ai_network

  # ── Grafana 시각화 대시보드 ──────────────────────────
  grafana:
    image: grafana/grafana:latest
    container_name: grafana
    restart: unless-stopped
    ports:

    environment:

      - GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_PASSWORD:-admin}
    volumes:
      - grafana_data:/var/lib/grafana   # 대시보드 설정 영속화
      - ./monitoring/grafana/dashboards:/etc/grafana/provisioning/dashboards  # 사전 정의 대시보드
    networks:
      - ai_network

volumes:                       # named volume 선언(컨테이너와 독립적으로 데이터 유지)
  model_cache:
    driver: local
  prometheus_data:
  grafana_data:

networks:

    driver: bridge
```

### (1) AWS EC2 인스턴스에 Docker 이미지 배포

AWS EC2(Elastic Compute Cloud)는 가상 서버를 임대하는 서비스로, AI API 배포에 가장 널리 사용되는 클라우드 플랫폼입니다. GPU 인스턴스(p3, g4dn, g5 패밀리)를 선택하면 고성능 AI 추론이 가능합니다.

| EC2 인스턴스 타입 | GPU | VRAM | vCPU/RAM | 시간당 비용 (USD) | 적합한 모델 |
| --- | --- | --- | --- | --- | --- |
| g4dn.xlarge | NVIDIA T4 | 16GB | 4 vCPU / 16GB | ~$0.526 | SD v1.5, LLaVA-7B (fp16) |
| g4dn.2xlarge | NVIDIA T4 | 16GB | 8 vCPU / 32GB | ~$0.752 | SD v2, ControlNet |
| g5.xlarge | NVIDIA A10G | 24GB | 4 vCPU / 16GB | ~$1.006 | SDXL, LLaVA-13B |
| p3.2xlarge | NVIDIA V100 | 16GB | 8 vCPU / 61GB | ~$3.060 | 대형 배치 추론 |
| t3.medium (CPU) | N/A | N/A | 2 vCPU / 4GB | ~$0.0416 | CLIP, 경량 모델 |

```bash
# ── Step 1: EC2 인스턴스 생성 (AWS CLI) ─────────────────
# AWS CLI 설치 및 설정
pip install awscli
aws configure    # Access Key, Secret Key, 기본 Region 등록(이후 명령 인증에 사용)

# 보안 그룹 생성(인스턴스의 방화벽 규칙 집합)
aws ec2 create-security-group \
  --group-name ai-api-sg \
  --description "AI API Security Group"

# 인바운드(외부→서버) 허용 규칙 추가
SG_ID="sg-xxxxxxxxxxxxxxxxx"    # 위 명령이 반환한 보안 그룹 ID로 교체

# --cidr 0.0.0.0/0 = 모든 IP 허용(운영에서는 IP 대역 제한 권장)
aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID --protocol tcp --port 22 --cidr 0.0.0.0/0   # SSH(원격 접속)

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID --protocol tcp --port 80 --cidr 0.0.0.0/0   # HTTP

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID --protocol tcp --port 443 --cidr 0.0.0.0/0  # HTTPS
```

### (2) SSH, 포트, 방화벽, Nginx 연동 실습

운영 환경에서 FastAPI 서비스는 직접 외부에 노출하지 않고 Nginx를 리버스 프록시로 앞에 두는 것이 표준 아키텍처입니다. Nginx가 HTTPS 종료(TLS Termination), 요청 제한(Rate Limiting), 정적 파일 서빙을 담당하고, FastAPI는 API 로직에만 집중합니다.

```nginx
# /etc/nginx/nginx.conf — Nginx 리버스 프록시 설정
user nginx;                  # 워커 프로세스 실행 계정
worker_processes auto;       # CPU 코어 수에 맞춰 워커 프로세스 자동 결정

events {
    worker_connections 1024; # 워커 1개당 동시 연결 수
}

http {
    include      /etc/nginx/mime.types;        # 확장자→MIME 타입 매핑
    default_type application/octet-stream;

    # ── 보안 헤더 (always: 에러 응답에도 적용) ─────────
    add_header X-Frame-Options        "SAMEORIGIN"    always;  # 클릭재킹 방지(iframe 제한)
    add_header X-Content-Type-Options "nosniff"       always;  # MIME 스니핑 방지
    add_header X-XSS-Protection        "1; mode=block" always;  # XSS 필터
    add_header Referrer-Policy         "strict-origin" always;  # 리퍼러 정보 최소화

    # ── 요청 크기 제한 ────────────────────────────────
    client_max_body_size 25M;    # 업로드 본문 최대 25MB(이미지 허용)
    client_body_timeout 120s;    # 본문 수신 타임아웃

    # ── 압축 설정 ─────────────────────────────────────
    gzip on;                                   # 응답 gzip 압축 활성화
    gzip_types application/json text/plain;    # 압축할 콘텐츠 타입

    # ── Rate Limiting ─────────────────────────────────
    # 클라이언트 IP별로 초당 10요청 제한, 상태 저장 영역 10MB 할당
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    # 백엔드(FastAPI 컨테이너) 풀 정의
    upstream fastapi_backend {
        server api:8000;        # Compose 서비스명 api의 8000 포트
        keepalive 32;           # 백엔드와의 연결을 32개까지 재사용
    }

    # ── HTTP → HTTPS 리다이렉트 ──────────────────────
    server {
        listen 80;
        server_name myapp.com www.myapp.com;
        return 301 https://$server_name$request_uri;   # 301 영구 이동으로 HTTPS 유도
    }

    # ── HTTPS 서버 ────────────────────────────────────
    server {
        listen 443 ssl http2;   # TLS + HTTP/2 활성화
        server_name myapp.com www.myapp.com;

        ssl_certificate     /etc/nginx/ssl/fullchain.pem;   # 인증서 체인
        ssl_certificate_key /etc/nginx/ssl/privkey.pem;     # 개인키
        ssl_protocols       TLSv1.2 TLSv1.3;                # 구버전 TLS 차단
        ssl_ciphers         HIGH:!aNULL:!MD5;               # 강한 암호 스위트만 허용

        # ── API 프록시 ────────────────────────────────
        location /api/ {
            # 위 zone 적용. burst=20: 순간 20개까지 버퍼, nodelay: 지연 없이 즉시 처리
            limit_req zone=api_limit burst=20 nodelay;
            proxy_pass          http://fastapi_backend;     # 백엔드로 요청 전달
            proxy_set_header    Host              $host;            # 원래 호스트 전달
            proxy_set_header    X-Real-IP         $remote_addr;     # 실제 클라이언트 IP
            proxy_set_header    X-Forwarded-For   $proxy_add_x_forwarded_for;  # 프록시 체인 IP
            proxy_set_header    X-Forwarded-Proto $scheme;          # http/https 구분 전달
            proxy_read_timeout  120s;     # 백엔드 응답 대기(추론 시간 고려)
            proxy_send_timeout  120s;
            proxy_request_buffering off;  # 본문을 버퍼링 없이 스트리밍(대용량 업로드)
        }

        # ── Swagger UI 접근 ───────────────────────────
        location /docs {
            proxy_pass http://fastapi_backend;   # API 문서 페이지 프록시
            proxy_set_header Host $host;
        }

        # ── 헬스체크 ──────────────────────────────────
        location /health {
            proxy_pass http://fastapi_backend;
            access_log off;          # 잦은 헬스체크는 로그에서 제외
        }
    }
}
```

### (4)(5) 토큰/모델 캐싱 이슈 해결 및 GitHub 자동 배포

클라우드 배포 시 가장 자주 발생하는 문제들과 GitHub Actions를 이용한 CI/CD 자동 배포 파이프라인 구성 방법을 다룹니다.

#### 배포 시 자주 발생하는 문제와 해결 방법

**문제 1: "Repository not found" — HF 토큰 미설정**
해결: 환경변수 `HUGGINGFACE_HUB_TOKEN` 또는 `HF_TOKEN` 설정
`export HF_TOKEN=hf_xxxx` 또는 `.env` 파일에 추가

**문제 2: 컨테이너 재시작마다 모델 재다운로드**
해결: Docker 볼륨으로 `HF_HOME` 캐시 디렉토리를 영구 마운트
`-v model_volume:/root/.cache/huggingface`

**문제 3: "No space left on device" — 모델 파일이 디스크를 가득 채움**
해결: EC2 볼륨 크기 증가 (`aws ec2 modify-volume`) 또는
`/tmp` 대신 별도 EBS 볼륨을 `/data`에 마운트

**문제 4: 첫 요청이 매우 느림 (Warm-up 문제)**
해결: 앱 시작 시 워밍업 추론 실행 (lifespan에서 더미 추론)
또는 Gunicorn `worker_init` 훅에서 모델 사전 로드

**문제 5: GPU OOM (Out of Memory)**
해결: 워커 수 감소, fp16/int4 양자화 적용, attention_slicing 활성화

```yaml
# .github/workflows/deploy.yml — GitHub Actions CI/CD 파이프라인
name: AI API Deploy to AWS EC2

on:                     # 워크플로 실행 트리거
  push:
    branches: [main]   # main 브랜치 푸시 시 자동 배포
  workflow_dispatch:    # Actions 화면에서 수동 실행 허용

env:                    # 모든 job이 공유하는 환경변수(secrets로 민감값 주입)
  DOCKER_IMAGE: ${{ secrets.DOCKERHUB_USERNAME }}/ai-fastapi
  EC2_HOST: ${{ secrets.EC2_PUBLIC_IP }}

jobs:
  # ── Step 1: 코드 테스트 ─────────────────────────────
  test:
    runs-on: ubuntu-latest        # 실행 러너 OS
    steps:
      - uses: actions/checkout@v4 # 저장소 코드 체크아웃

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: pip              # pip 캐시로 의존성 설치 가속

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest httpx   # 테스트 도구

      - name: Run tests
        run: pytest tests/ -v --tb=short   # 테스트 실행(--tb=short: 간결한 트레이스백)

  # ── Step 2: Docker 이미지 빌드 및 푸시 ──────────────
  build-and-push:
    needs: test                   # test job 성공 후에만 실행
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Log in to Docker Hub
        uses: docker/login-action@v3   # Docker Hub 인증
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3   # 캐시 지원 빌더 활성화

      - name: Build and push image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true                          # 빌드 후 레지스트리로 푸시
          tags: |                             # 두 개 태그로 푸시(latest + 커밋 SHA)
            ${{ env.DOCKER_IMAGE }}:latest
            ${{ env.DOCKER_IMAGE }}:${{ github.sha }}
          cache-from: type=gha                # GitHub Actions 캐시에서 레이어 재사용
          cache-to: type=gha,mode=max         # 빌드 레이어를 캐시에 저장

  # ── Step 3: EC2 서버 배포 ────────────────────────────
  deploy:
    needs: build-and-push                     # 이미지 푸시 성공 후 실행
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to EC2
        uses: appleboy/ssh-action@v1          # SSH로 원격 서버에 명령 실행
        with:
          host: ${{ env.EC2_HOST }}
          username: ubuntu
          key: ${{ secrets.EC2_SSH_PRIVATE_KEY }}   # SSH 개인키(secrets)
          script: |                            # 서버에서 순차 실행할 배포 스크립트
            cd /home/ubuntu/ai-fastapi
            docker compose pull api            # 방금 푸시한 새 이미지 받기
            docker compose up -d --no-deps api # api만 무중단 교체(의존 서비스는 유지)
            docker system prune -f --filter "until=24h"  # 24시간 지난 미사용 리소스 정리
            echo "배포 완료: $(date)"
```

### (1) 요청/응답 로그 남기기 (LoggingMiddleware)

운영 중인 API에서 발생하는 요청과 응답을 체계적으로 로깅하는 것은 장애 추적, 성능 분석, 보안 감사에 필수적입니다. FastAPI의 미들웨어(Middleware) 기능을 활용하면 모든 요청에 대해 일관된 로그를 남길 수 있습니다.

```python
# logging_middleware.py — 요청/응답 로깅 미들웨어
import time, uuid, json
from fastapi import Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
from loguru import logger


class RequestLoggingMiddleware(BaseHTTPMiddleware):
    """모든 HTTP 요청/응답을 구조화된 JSON으로 로깅합니다."""

    # 로깅 제외 경로 (자주 호출되는 헬스체크/메트릭은 로그 노이즈라 제외)
    SKIP_PATHS = {"/health", "/metrics", "/favicon.ico"}

    # dispatch: 모든 요청이 거쳐 가는 미들웨어 진입점
    async def dispatch(self, request: Request, call_next) -> Response:
        # 제외 경로면 로깅 없이 그대로 통과
        if request.url.path in self.SKIP_PATHS:
            return await call_next(request)

        # 요청마다 고유 ID 생성(앞 8자리) → 로그·응답 헤더로 추적
        request_id = str(uuid.uuid4())[:8]
        request.state.request_id = request_id   # 핸들러에서도 재사용 가능

        # 처리 시간 측정 시작
        start_time = time.time()
        # 실제 클라이언트 IP 추출: 프록시 헤더 우선, 없으면 직접 연결 IP
        client_ip = request.headers.get("X-Real-IP") or \
            request.headers.get("X-Forwarded-For", "").split(",")[0].strip() or \
            (request.client.host if request.client else "unknown")

        # 공통 로그 필드 수집
        log_data = {
            "request_id": request_id,
            "method":     request.method,
            "path":       request.url.path,
            "query":      str(request.url.query) or None,
            "client_ip":  client_ip,
            "user_agent": request.headers.get("User-Agent", "")[:100],  # 과도한 길이 방지
        }

        # 실제 요청 처리(다음 미들웨어/핸들러 호출)
        try:
            response = await call_next(request)
            status_code = response.status_code
        except Exception as exc:
            # 예외 발생 시에도 소요시간·에러를 로깅 후 재발생(전파)
            elapsed_ms = (time.time() - start_time) * 1000
            logger.error({
                **log_data,
                "status": 500,
                "elapsed_ms": round(elapsed_ms, 2),
                "error": str(exc)
            })
            raise

        # 응답까지 걸린 시간(ms) 계산
        elapsed_ms = (time.time() - start_time) * 1000
        log_entry = {
            **log_data,
            "status": status_code,
            "elapsed_ms": round(elapsed_ms, 2),
        }

        # 상태 코드/지연 시간에 따라 로그 레벨 분류(운영 알림 우선순위에 활용)
        if status_code >= 500:
            logger.error(json.dumps(log_entry, ensure_ascii=False))      # 서버 오류
        elif status_code >= 400:
            logger.warning(json.dumps(log_entry, ensure_ascii=False))    # 클라이언트 오류
        elif elapsed_ms > 5000:
            logger.warning(json.dumps({**log_entry, "slow_request": True}, ensure_ascii=False))  # 5초 초과 느린 요청
        else:
            logger.info(json.dumps(log_entry, ensure_ascii=False))       # 정상

        # 응답 헤더에 추적 정보 부착(클라이언트/디버깅용)
        response.headers["X-Request-ID"] = request_id
        response.headers["X-Response-Time"] = f"{elapsed_ms:.2f}ms"
        return response


# ── loguru 설정 (main.py에서 호출) ──────────────────────
from loguru import logger
import sys


def setup_logging(log_file: str = "./logs/app.log"):
    """loguru 핸들러 설정"""
    import os
    os.makedirs(os.path.dirname(log_file), exist_ok=True)   # 로그 디렉터리 보장

    # loguru 기본 핸들러 제거(중복 출력 방지)

    logger.add(sys.stderr, level="DEBUG" if os.getenv("DEBUG") else "INFO",

               serialize=True)        # 한 줄을 JSON으로 직렬화(로그 수집기 연동)
```

### (2) Uvicorn/Gunicorn 로그 설정

Uvicorn과 Gunicorn의 로그 설정을 통해 서버 레벨의 액세스 로그와 에러 로그를 체계적으로 관리합니다. JSON 형식의 구조화 로그는 Elasticsearch, Loki 등 로그 수집 시스템과 쉽게 통합됩니다.

```python
# uvicorn_logging.py — Uvicorn JSON 구조화 로그 설정
import logging
import json
from datetime import datetime


class UvicornJSONFormatter(logging.Formatter):
    """Uvicorn 로그를 JSON 형식으로 출력하는 포매터"""

    # format: 각 로그 레코드를 JSON 문자열로 변환
    def format(self, record: logging.LogRecord) -> str:
        log_data = {
            "timestamp": datetime.utcnow().isoformat() + "Z",  # UTC ISO8601 + Z 표기
            "level":     record.levelname,    # 로그 레벨
            "logger":    record.name,         # 로거 이름
            "message":   record.getMessage(), # 실제 메시지
        }
        if record.exc_info:   # 예외가 있으면 트레이스백도 포함
            log_data["exception"] = self.formatException(record.exc_info)
        return json.dumps(log_data, ensure_ascii=False)   # 한글 깨짐 방지
```

### (3)(4) 응답 속도 측정 및 개선 전략

AI API의 응답 속도는 사용자 경험에 직결됩니다. 느린 응답의 원인을 정확히 파악하고 단계별로 최적화하는 체계적인 접근이 필요합니다.

```python
# performance_profiler.py — 엔드포인트 성능 측정 도구
import time, asyncio
import functools
from fastapi import Request
from loguru import logger


# ── 데코레이터 기반 함수 수준 프로파일링 ─────────────────
def profile_async(func):
    """비동기 함수의 실행 시간을 측정하는 데코레이터"""
    @functools.wraps(func)   # 원본 함수의 이름/독스트링 보존
    async def wrapper(*args, **kwargs):
        start = time.perf_counter()   # 고정밀 타이머 시작
        result = await func(*args, **kwargs)   # 실제 함수 실행
        elapsed = (time.perf_counter() - start) * 1000   # 소요 ms
        logger.debug(f"[PROFILE] {func.__name__}: {elapsed:.2f}ms")
        if elapsed > 1000:   # 1초 초과 시 경고로 승격
            logger.warning(f"[SLOW] {func.__name__}: {elapsed:.2f}ms 초과")
        return result
    return wrapper


# ── 단계별 시간 측정 컨텍스트 매니저 ─────────────────────
class Timer:
    """코드 블록의 실행 시간을 측정합니다."""

    def __init__(self, label: str = ""):
        self.label = label
        self.checkpoints = {}    # 구간별 누적 시간 저장
        self._start = None

    def start(self):
        self._start = time.perf_counter()   # 측정 시작 시각 기록
        return self                          # 체이닝 가능(Timer().start())

    def checkpoint(self, name: str):
        """시작 시점부터 현재까지 누적 시간을 name으로 기록"""
        self.checkpoints[name] = (time.perf_counter() - self._start) * 1000

    def stop(self) -> dict:
        # 전체 소요시간과 각 체크포인트를 요약 dict로 반환
        total = (time.perf_counter() - self._start) * 1000
        return {
            "label": self.label,
            "total_ms": round(total, 2),
            "checkpoints": {k: round(v, 2) for k, v in self.checkpoints.items()}
        }


# ── 실제 사용 예시 ───────────────────────────────────────
@app.post("/api/v1/generate")
async def generate_image(request: ImageGenRequest):
    timer = Timer("image_generation").start()   # 타이머 시작

    # 단계 1: 입력 검증
    validated = validate_request(request)
    timer.checkpoint("validation")      # 여기까지 누적 시간 기록

    # 단계 2: 전처리
    processed = preprocess(validated)
    timer.checkpoint("preprocessing")

    # 단계 3: 모델 추론 (보통 가장 오래 걸리는 병목 구간)
    result = await run_inference(processed)
    timer.checkpoint("inference")

    # 단계 4: 후처리
    response = postprocess(result)
    timer.checkpoint("postprocessing")

    timing = timer.stop()                # 단계별 시간 요약 산출
    logger.info(f"타이밍: {timing}")     # 어느 구간이 느린지 로그로 확인
    # 출력 예: {"label":"image_generation","total_ms":3420.15,
    # "checkpoints":{"validation":2.1,"preprocessing":45.3,
    # "inference":3350.2,"postprocessing":22.5}}

    return response
```

```python
def setup_prometheus(app: FastAPI) -> None:
    """FastAPI 앱에 Prometheus 계측기를 부착합니다."""

    instrumentator = Instrumentator(
        should_group_status_codes=False,   # 200/201 등을 묶지 않고 개별 집계
        should_respect_env_var=True,       # 환경변수로 메트릭 on/off 제어

    instrumentator.add(metrics.request_size())
    instrumentator.add(metrics.response_size())
    instrumentator.add(metrics.latency(buckets=[0.1, 0.5, 1, 5, 10, 30]))

    # 앱 계측 시작 + /metrics 엔드포인트로 노출(Prometheus가 스크랩)
    instrumentator.instrument(app).expose(app, endpoint="/metrics")


# ── GPU 메트릭 주기적 업데이트 ──────────────────────────
async def update_gpu_metrics():
    """GPU 메모리 사용량을 주기적으로 Prometheus에 기록합니다."""
    while True:
        if torch.cuda.is_available():
            for i in range(torch.cuda.device_count()):   # 모든 GPU 순회

        await asyncio.sleep(15)     # 15초마다 반복(비동기 sleep)
```

```python
from monitoring import MODEL_INFERENCE_TIME, MODEL_REQUESTS_TOTAL, ACTIVE_REQUESTS


@app.post("/api/v1/generate")
async def generate_with_metrics(request: ImageGenRequest):
    ACTIVE_REQUESTS.inc()         # 동시 처리 요청 수 +1
    start = time.time()
    try:
        result = await do_inference(request)
        MODEL_REQUESTS_TOTAL.labels(   # 성공 카운터 증가
            model_name="stable-diffusion", status="success"
        ).inc()
        return result
    except Exception as e:
        MODEL_REQUESTS_TOTAL.labels(   # 실패 카운터 증가
            model_name="stable-diffusion", status="error"
        ).inc()
        # raise하면 미들웨어/Sentry가 에러를 자동 포착·전송
        raise
    finally:
        elapsed = time.time() - start
        MODEL_INFERENCE_TIME.observe(elapsed)   # 추론 시간 히스토그램에 기록
        ACTIVE_REQUESTS.dec()                   # 처리 끝났으니 -1(성공/실패 무관)
```
